## 1. ELO-система рейтинга менеджеров

### 1.1 Почему ELO, а не просто баллы?

Традиционная кумулятивная система баллов имеет фатальный недостаток: менеджер, работающий год, всегда будет лидером рейтинга, а новый менеджер его никогда не догонит. ELO решает это:

- ELO = **относительный рейтинг**, основанный на **текущей** эффективности
- Менеджер «играет матч» каждый день — против своего ожидаемого результата
- Если работает лучше ожидаемого → рейтинг растёт
- Если хуже → падает
- **Новый менеджер может обогнать «ветерана», если работает эффективнее**

### 1.2 Математическая модель

```
ELO_new = ELO_old + K × (Score - Expected)
```

**Компоненты:**

| Параметр | Значение | Объяснение |
|----------|---------|----------|
| `ELO_start` | 1000 | Начальный рейтинг для нового менеджера |
| `K` (новичок, < 30 дней) | 40 | Быстрая адаптация рейтинга |
| `K` (средний, 30-90 дней) | 32 | Стандартная скорость изменения |
| `K` (опытный, > 90 дней) | 24 | Стабильный рейтинг, меньше колебаний |

**Score (фактический результат дня):**
```python
def calculate_daily_score(metrics):
    # Нормализация каждого компонента к 0-1
    clients_score = min(1.0, processed / 50)         # 50 = норма
    conversion_score = success_rate * 2               # конверсия × 2
    followup_score = 1.0 - missed_followup_rate       # дисциплина
    shop_score = min(1.0, shops_created / 0.4)        # 0.4 = 2 в неделю / 5 дней
    revenue_score = min(1.0, daily_revenue / target_revenue)
    
    # Взвешенная сумма
    score = (
        clients_score * 0.25 +
        conversion_score * 0.30 +
        followup_score * 0.15 +
        shop_score * 0.20 +
        revenue_score * 0.10
    )
    return min(1.0, max(0.0, score))
```

**Expected (ожидаемый результат):**
```python
def calculate_expected(elo_manager, elo_average=1000):
    # Классическая ELO формула
    return 1.0 / (1.0 + 10 ** ((elo_average - elo_manager) / 400))
```

### 1.4 Механика decay (затухания)

```python
# Ежедневный decay при отсутствии активности
if no_activity_today:
    elo -= 5  # -5 ELO за каждый пропущенный рабочий день

# Еженедельный decay если не подключены магазины
if shops_connected_this_week == 0 and worked_days >= 3:
    elo -= 15  # Серьёзный штраф

# Floor (минимальный ELO)
elo = max(500, elo)  # Нельзя упасть ниже 500
```

### 1.5 Защита от абуза

1. **ELO не считается за дни без минимальной активности** (< 1 час на сайте)
2. **K-factor уменьшается при подозрительной активности** (100 контактов, 0 конверсий)
3. **Admin override**: администратор может заморозить ELO при расследовании
4. **Weekly reset cap**: максимум ±50 ELO за неделю (защита от outlier-дней)

### 2.2 Progressive баллы (прогрессивная система)

```python
# За каждый следующий контакт в день — множитель
def progressive_multiplier(contact_number_today):
    if contact_number_today <= 10:
        return 1.0      # Первые 10 — стандарт
    elif contact_number_today <= 30:
        return 1.1      # 11-30 — бонус +10%
    elif contact_number_today <= 50:
        return 1.2      # 31-50 — бонус +20%
    else:
        return 0.8      # 51+ — снижение (защита от data stuffing)
```

### 3.3 Формула нормализации

```python
def normalize_kpd_to_100(raw_kpd, max_kpd=5.0):
    """
    Текущий КПД: 0-5.0; Новый: 0-100.
    Логарифмическая кривая для лучшей разрешающей способности
    в среднем диапазоне.
    """
    if raw_kpd <= 0:
        return 0
    # Логарифмическое масштабирование
    normalized = (raw_kpd / max_kpd) ** 0.85 * 100
    return min(100, max(0, round(normalized)))
```

### 3.4 Дневной/Недельный/Месячный спидометр

Менеджер видит **три спидометра**:
1. **Сегодня** — текущий КПД за день (обновляется в реальном времени)
2. **Эта неделя** — средний КПД за рабочие дни
3. **Этот месяц** — тренд КПД за 30 дней (с графиком)

**Стрелки тренда:**
- ⬆️ Зелёная стрелка: КПД растёт (> +5% от вчера)
- ➡️ Серая стрелка: КПД стабилен (±5%)
- ⬇️ Красная стрелка: КПД падает (< -5% от вчера)

### 4.2 Справедливый рейтинг (не кумулятивный)

**Проблема**: Менеджер, работающий месяц, всегда впереди.

**Решение**: Рейтинг основан на **ELO**, а не на сумме баллов.

Дополнительно:
- Недельный рейтинг: ELO-дельта за неделю (кто больше вырос)
- Месячный рейтинг: средний ежедневный КПД
- All-time: средний ELO за последние 30 дней

### 4.3 Командные соревнования

Если менеджеров много (> 6), можно делить на «команды» по 2-3 человека:
- Командный ELO = средний ELO участников
- Еженедельное соревнование между командами
- Победившая команда получает бонус (+50 грн каждому)

## 5. Привязка к зарплате

### 5.1 Структура компенсации

```
ЗАРПЛАТА МЕНЕДЖЕРА = Ставка + Комиссия + Бонусы - Штрафы

Ставка (дневная):
    КПД >= 50/100 → полная ставка за день
    КПД 30-49     → 50% ставки
    КПД < 30      → 0 ставки

Комиссия:
    % от оплаченных накладных (WholesaleInvoice.payment_status = "paid")
    Процент зависит от ELO тира (3%-12%)
    
Бонусы:
    ≥ 2 магазина подключено за неделю → +500 грн
    ТОП-1 за неделю → +300 грн
    Все отчёты вовремя за месяц → +200 грн

Штрафы:
    0 магазинов за неделю при 3+ раб. днях → -200 грн
    > 20% пропущенных перезвонов → -100 грн
```

### 5.2 KPI контракт менеджера

| KPI | Норма | Minimum | Отличный |
|-----|-------|---------|----------|
| Звонки/день | 50 | 30 | 70+ |
| Магазины/неделя | 2 | 1 | 4+ |
| КПД/день | 60 | 45 | 80+ |
| Перезвоны выполнены | 90% | 80% | 98%+ |
| Отчёт вовремя | Да | - | - |
| Revenue/месяц | 50K грн | 25K | 100K+ |

### 5.3 Автоматический расчёт

```python
def calculate_daily_salary(manager, date):
    kpd = get_normalized_kpd(manager, date)  # 0-100
    daily_rate = settings.MANAGER_DAILY_RATE  # напр. 500 грн
    
    if kpd >= 50:
        salary = daily_rate
    elif kpd >= 30:
        salary = daily_rate * 0.5
    else:
        salary = 0
    
    # Менеджер видит в реальном времени:
    # "Сьогодні ваш КПД = 42. Якщо не дожмете до 50, 
    #  ви не отримаєте ставку за сьогодні."
    return salary
```

## 7. Психологические исследования

### 7.1 Теория самоопределения (Self-Determination Theory — Deci & Ryan)

Мотивация сотрудника базируется на трёх потребностях:
1. **Автономия** — менеджер сам выбирает, кому звонить, какие источники использовать
2. **Компетентность** — прогрессивная система баллов показывает рост навыков
3. **Связь с другими** — лидерборд и командные соревнования

**Применение**: Не навязывать жёсткий порядок работы, но давать понятную обратную связь о результатах.

### 7.2 Theory of Goal Setting (Locke & Latham)

Ключевые принципы:
- **Конкретные цели** лучше «старайся больше» → «50 звонков, 2 магазина»
- **Сложные, но достижимые** → КПД 60 как минимум, 80 как отличный
- **Обратная связь** → real-time спидометр
- **Commitment** → менеджер сам видит привязку к зарплате

### 7.3 Gamification Psychology

Исследования показывают:
- **Points + Leaderboards + Badges** увеличивают продуктивность на **15-20%** (Deterding et al., 2011)
- **Loss aversion** (Kahneman & Tversky): люди сильнее мотивированы **избежать потери**, чем получить награду → «Ваш КПД падает, ви ризикуєте не отримати ставку» эффективнее, чем «Заработайте бонус»
- **Variable reward**: непредсказуемые бонусы (random bonus day) — на **40% мотивирующее**, чем фиксированные
- **Social comparison**: видеть рейтинг относительно коллег — мощный мотиватор (но может демотивировать отстающих, поэтому нужны «улучшение» бейджи)

### 7.4 Flow State (Mihaly Csikszentmihalyi)

Менеджер входит в «поток» когда:
- Задача не слишком простая (скучно) и не слишком сложная (стресс)
- Есть чёткие цели и немедленная обратная связь
- **Применение**: real-time обновление КПД, toast-нотификации при баллах

### Пересмотр ELO в контексте `compute_kpd()`

> [!IMPORTANT]
> **Молекулярный анализ v3 показал**: текущая KPD формула уже содержит элементы, схожие с ELO:
> - **4-компонентная система** (Effort, Quality, Ops, Penalty) — аналог ELO factor decomposition
> - **Bayesian smoothing** `(SW+1)/(N+5)` — защита от малых выборок, как K-factor в ELO
> - **Субквадратные кривые** (^0.6, ^0.55) — diminishing returns, как в рейтинговых системах
> - **DB-configurable веса** — можно менять без деплоя

**Вывод**: Не нужно заменять KPD на ELO. Лучше **расширить KPD** элементами ELO:

```python
# Предложенный гибрид: KPD + ELO delta
class HybridRating:
    def __init__(self):
        self.kpd = compute_kpd(metrics, config)  # Текущий (сохранить!)
        self.elo_base = 1500  # Стартовый ELO
        self.k_factor = 32    # Чувствительность

    def update_elo(self, kpd_current, kpd_expected):
        """ELO delta = K × (actual_kpd/max_kpd - expected)"""
        actual_score = min(1.0, kpd_current['value'] / 4.0)
        expected_score = min(1.0, kpd_expected['value'] / 4.0)
        delta = self.k_factor * (actual_score - expected_score)
        self.elo_base += delta
        return self.elo_base
```
