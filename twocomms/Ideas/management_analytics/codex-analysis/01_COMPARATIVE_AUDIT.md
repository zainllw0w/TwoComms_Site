# Comparative Audit: Existing AI Docs vs Codex Standard

## 1. Контекст аудита
Объект сравнения:
- Текущий набор `twocomms/Ideas/management_analytics/01..08`.
- Новый пакет `codex-analysis`.

Цель:
- Сохранить лучшие идеи существующих документов.
- Убрать несостыковки между идеями и реальным кодом.
- Дать единый implementation-ready стандарт.

## 2. Что в текущих документах сделано сильно
1. Глубокая проработка HES, provisional points и anti-gaming.
2. Хорошее внимание к follow-up дисциплине и повторным продажам.
3. Сильный фокус на дедупликации и email-логировании КП.
4. Корректный бизнес-принцип: итоговая ценность = оплаченная конверсия.

## 3. Что в текущих документах проблемно
1. Смешение уровней: стратегия, UX идеи, модели и roadmap иногда перемешаны.
2. Есть места, где описаны уязвимости, уже частично закрытые в коде.
3. Недостаточно формализована trust-логика при отсутствии записей звонков.
4. В DTF предложениях местами закладывались write-сценарии без ограничений multi-DB.

## 4. Ground truth из фактического кода (на момент аудита)
1. Уже реализованы: KPD, advice engine, follow-up sync, CP email logs.
2. В `management/views.py` при создании клиента из `home()` строгая дедупликация до создания не обязательна.
3. DTF маршрутизируется отдельным router; возможна отдельная БД `dtf`.
4. Celery и beat присутствуют, но alerting pipeline нужно расширять.

## 5. Принципы нового стандарта
1. Decision-complete: ни один ключевой параметр не остается "на усмотрение" имплементатора.
2. Проверяемость: для каждой формулы и правила есть тест-критерии.
3. Anti-abuse by design: self-reported события не могут доминировать над hard results.
4. Role separation: manager и admin видят разные уровни аналитики.
5. Integration safety: DTF v1 только read-only bridge.

## 6. Матрица сравнения
| Блок | Existing 01..08 | Codex standard |
|---|---|---|
| Формула эффективности | Сильная, но местами вариативная | Фиксированная гибридная модель с gate/trust/ROI/ELO |
| Anti-abuse | Много идей, частично дублируются | Единый trust слой и детальная матрица атак |
| KPI/оплата | Описано | Сведено в формальный unit economics contract |
| Alerts | Идеи + часть текущей реализации | Полный event-schedule и no-report политика |
| DTF | Есть идеи write/manage | V1 read-only с multi-DB guardrails |
| Тестируемость | Частично | Полная acceptance matrix |

## 7. Что переносим без изменений
1. Акцент на оплаченной конверсии как главном результате.
2. 3 попытки по no-answer и обязательные follow-up цепочки.
3. Эскалационные напоминания и ежедневные советы.
4. Отдельность DTF статистики от brand-статистики.

## 8. Что перерабатываем
1. Формулы в единую гибридную модель (без конкурирующих версий в разных файлах).
2. Привязку score к trust tiers.
3. DTF архитектуру под read-only v1 и cross-DB ограничения.
4. Стандарты API-контрактов и тестов.

## 9. Итог
Новый пакет не противоречит сильным частям старых документов, но устраняет неоднозначность и делает спецификацию пригодной для прямой реализации и объективного сравнения AI-этапов.
