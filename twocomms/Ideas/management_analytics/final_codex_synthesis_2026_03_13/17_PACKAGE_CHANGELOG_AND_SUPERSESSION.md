# Package Changelog And Supersession

## What This Package Supersedes

This package supersedes `final_codex_synthesis_2026_03_12` as the main canonical handoff package for implementation planning.

The old baseline remains valuable as a source layer, but no longer as the primary package the next agent should read first.

## Why Supersession Was Necessary

The 2026-03-12 package was strong, but still required the reader to combine:
- baseline authoritative docs;
- `tc_mosaic_improvements` deltas;
- `i1` code-reality and readiness audits.

That created three recurring risks:
- source-of-truth drift;
- missed code-reality constraints;
- re-synthesis burden for the next agent.

## Main Improvements In 2026-03-13

### 1. Authority hardening

- added explicit authority manifest;
- separated canonical, superseded and reference layers;
- added source coverage matrix.

### 2. Code-reality hardening

- integrated existing model/service/template reality into the main package;
- marked extend vs keep vs create;
- elevated current KPD, shop subsystem, CP logs, Telegram workflows and `ManagementStatsConfig` to first-class planning constraints.

### 3. Fairness hardening

- added `capacity_factor` and reintegration logic;
- made `score_confidence` a shared contract;
- added shared-phone and import burst protections;
- strengthened tiny-team benchmark and stale/freshness semantics.

### 4. Implementation hardening

- added testing strategy;
- restored single-source defaults registry convenience;
- added migration/governance/job contract;
- added implementation blueprint;
- added dedicated handoff brief.

## What Stayed Stable

- evolution over current `management`, not rewrite;
- shared-hosting stack;
- shadow-first rollout;
- verified money truth over shadow analytics;
- no telephony/QA payroll dependence before maturity;
- no DTF mixing into wholesale truth.

## Why The New Package Is Easier For The Next Agent

- fewer ambiguous files;
- more explicit authority boundaries;
- one place for code-reality constraints;
- one place for build order and verification;
- one place for anti-loss traceability.

## Residual Risks That Still Require Explicit Handling

- unresolved drift decisions like `xml_connected`;
- exact migration shapes for some new models;
- implementation-level service boundaries in code;
- production data backfill details.

These are now explicit and should enter the next implementation file as named decisions, not hidden gaps.
