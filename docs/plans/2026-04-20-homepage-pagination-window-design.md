# Дизайн окна пагинации на главной

## Статус

Решение согласовано в переписке 2026-04-20 на уровне spec и planning.

## Контекст

На главной странице сейчас одновременно используются два паттерна навигации по товарам:

- `Показати ще` догружает следующую страницу товаров в уже открытый список.
- Нижняя пагинация выводит все номера страниц из `paginator.page_range`.

Пока страниц мало, это выглядит терпимо. Но при `15+` страницах блок начинает разрастаться, перегружать интерфейс и снова создаёт давление на мобильную вёрстку. Недавний фикс мобильного масштаба решает переполнение, но не должен бесконечно ужимать растущий ряд кнопок.

## Проблема

Текущая пагинация на главной рендерит полный список всех страниц, из-за чего возникают три проблемы:

1. Ширина блока растёт линейно вместе с количеством страниц.
2. Визуальная модель начинает конфликтовать с ролью `Показати ще`, которая уже является основным действием для последовательного просмотра.
3. Интерфейс остаётся формально рабочим, но перестаёт быть удобным и понятным при большом количестве страниц.

## Внешние ориентиры

Решение ниже опирается на паттерны, поднятые через Context7:

- `Material UI` использует компактные окна пагинации через `boundaryCount` и `siblingCount`, то есть подход “края + текущая зона + многоточия”.
- `Ant Design` для большого числа страниц рекомендует compact/ellipsis-модель, а сверхупрощённый режим только для очень тесных интерфейсов.
- `Shopify Polaris` разделяет пагинацию и progressive disclosure: длинные списки должны оставаться digestible, а `load more` может жить рядом как отдельный сценарий.

Вывод у всех трёх один: длинный список всех номеров страниц не должен оставаться полностью развёрнутым, когда число страниц растёт.

## Цели

- Сделать пагинацию на главной компактной и стабильной при любом числе страниц.
- Сохранить прямые URL страниц для SEO и явного перехода на конкретную страницу.
- Оставить `Показати ще` основным действием для последовательного раскрытия каталога.
- Использовать один источник истины для окна пагинации, чтобы не разъезжались сервер и клиент.

## Не цели

- Не менять пагинацию каталога в рамках этой задачи.
- Не добавлять History API-синхронизацию для `Показати ще`; сохраняем текущее поведение, где progressive loading меняет состояние интерфейса, но не переписывает URL.
- Не переделывать визуальный стиль блока шире, чем это нужно для компактной пагинации и многоточий.

## Рассмотренные подходы

### 1. Всегда показывать все номера страниц

Плюсы:

- Минимальная стоимость реализации.
- Нет изменения поведения.

Минусы:

- Плохо масштабируется при `15+`, `30+` и далее.
- Заставляет мобильный блок либо ужиматься всё сильнее, либо снова скроллиться.
- Делает `Показати ще` визуально вторичным и частично лишним.

Решение:

- Отклонить.

### 2. Свести блок к `Назад / Текущая / Вперёд`

Плюсы:

- Очень компактно.
- Стабильно на любой ширине.

Минусы:

- Убирает быстрый прыжок на далёкую страницу.
- Слишком сильно упрощает UX относительно того, что уже есть.

Решение:

- Отклонить.

### 3. Компактное окно страниц с многоточиями

Плюсы:

- Держит блок маленьким и предсказуемым.
- Сохраняет переход на первую, последнюю и соседние страницы.
- Естественно сочетается с `Показати ще`.
- Совпадает с практиками крупных дизайн-систем.

Минусы:

- Добавляет логику расчёта видимого окна.
- Требует отдельного partial и небольшого обновления AJAX-ответа.

Решение:

- Принять.

## Итоговое решение

На главной используется компактная windowed pagination.

Правила:

- Если страниц `<= 7`, показываются все номера.
- Если страниц `>= 8`, показываются:
  - предыдущая страница,
  - первая страница,
  - текущая страница,
  - по одной соседней странице слева и справа, если они есть,
  - последняя страница,
  - многоточия между разорванными диапазонами,
  - следующая страница.

Рекомендуемые параметры:

- `expand_all_threshold = 7`
- `boundary_count = 1`
- `sibling_count = 1`

Примеры:

- страница `1` из `15`: `1 2 3 … 15`
- страница `8` из `15`: `1 … 7 8 9 … 15`
- страница `15` из `15`: `1 … 13 14 15`

`Показати ще` остаётся основным способом последовательного просмотра. Нижняя пагинация остаётся механизмом явного перехода на конкретный URL страницы.

## UX-правила

### Первичная загрузка страницы

- Главная рендерит компактную пагинацию на основе `page_obj.number` и `paginator.num_pages`.
- Активная страница сохраняет `aria-current="page"`.
- Многоточия неинтерактивны и не попадают в фокус.

### После `Показати ще`

- Товары следующей страницы добавляются как и сейчас.
- Текущая страница в состоянии интерфейса увеличивается.
- Нижняя пагинация пересобирается по тем же серверным правилам.
- Если страниц больше нет, `Показати ще` скрывается как сейчас.

### Прямой клик по пагинации

- Клик по номеру страницы ведёт на реальный URL этой страницы.
- Клик по `Попередня` и `Наступна` ведёт на соседний реальный URL.

## Технический дизайн

### Источник истины

Окно пагинации должно вычисляться только на backend.

Обоснование:

- `load_more_products` уже сейчас возвращает серверно отрендеренный HTML через `render_to_string`.
- Если сервер будет отдавать и `pagination_html`, то не придётся держать один и тот же алгоритм одновременно в Python и JavaScript.
- Это уменьшает риск расхождения логики и упрощает дальнейшую поддержку.

### Предлагаемая backend-структура

Добавить небольшой чистый helper, предпочтительно в отдельный модуль:

- `twocomms/storefront/pagination.py`

Предлагаемый API:

```python
build_homepage_pagination_items(
    *,
    current_page: int,
    total_pages: int,
    expand_all_threshold: int = 7,
    boundary_count: int = 1,
    sibling_count: int = 1,
) -> list[dict]
```

Каждый элемент должен иметь один из типов:

- `page`
- `ellipsis`
- `prev`
- `next`

Пример page-элемента:

```python
{
    "type": "page",
    "page": 8,
    "url": "/?page=8",
    "is_current": True,
}
```

Пример ellipsis-элемента:

```python
{
    "type": "ellipsis",
}
```

Для `prev` и `next` должны присутствовать:

- целевая страница,
- URL,
- признак disabled.

### Шаблон

Вынести разметку пагинации главной в отдельный partial, например:

- `twocomms/twocomms_django_theme/templates/partials/home_pagination.html`

Страница главной должна рендерить стабильный контейнер-обёртку:

- с id или data-атрибутом, предназначенным для замены блока пагинации после `load more`.

Основной шаблон должен перестать итерироваться по сырым `paginator.page_range`. Вместо этого он должен рендерить уже вычисленный список элементов пагинации.

### Обновление AJAX-ответа

Расширить JSON-ответ `load_more_products` полем:

- `pagination_html`

Это поле должно формироваться тем же partial-шаблоном пагинации на основе helper-результата для нового `current_page`.

### Frontend-путь обновления

`homepage.js` больше не должен владеть логикой построения окна страниц.

После успешного ответа `Показати ще` фронтенд должен:

1. добавить `products_html`,
2. обновить метаданные текущей и общей страницы,
3. заменить содержимое стабильного контейнера пагинации на `pagination_html`,
4. заново найти текущий DOM-элемент пагинации,
5. прогнать существующую логику mobile layout/sync для уже обновлённого DOM.

Так мы сохраняем один источник истины и при этом не ломаем уже существующую адаптивную логику.

## Влияние на стили

Существующий визуальный стиль можно в целом сохранить.

Потребуются только небольшие добавления:

- стили для неинтерактивных многоточий,
- проверка, что компактный блок так же корректно масштабируется на mobile,
- сохранение существующих активных и disabled-состояний.

Широкий редизайн не нужен.

## Стратегия тестирования

### Backend-тесты

Добавить точечные тесты для helper-функции:

- `1 / 5` возвращает все страницы,
- `1 / 15` возвращает `1 2 3 … 15`,
- `8 / 15` возвращает `1 … 7 8 9 … 15`,
- `15 / 15` возвращает `1 … 13 14 15`.

Добавить endpoint-покрытие для `load_more_products`:

- ответ содержит `pagination_html`,
- возвращённая пагинация отражает новый `current_page`.

### Тесты шаблона

Проверить, что главная рендерит пагинацию через компактный список элементов, а не через прямую итерацию полного `paginator.page_range`.

### Frontend-проверки

Проверить, что после замены HTML пагинации:

- правильно обновляется активная страница,
- mobile scaling hook всё ещё запускается,
- блок не выходит за экран на узкой ширине.

### Браузерная верификация

Минимально нужно проверить:

- что малое количество страниц выглядит нормально,
- что при большем количестве страниц получается компактное окно,
- что `Показати ще` корректно перестраивает нижнюю пагинацию,
- что mobile layout остаётся внутри viewport.

## Риски

### Риск: расхождение backend и frontend

Снижение риска:

- логика окна живёт только на backend,
- AJAX получает уже готовый серверный `pagination_html`.

### Риск: устаревшие JS-ссылки на DOM после замены блока

Снижение риска:

- использовать стабильную обёртку,
- после каждой замены DOM заново получать актуальный элемент пагинации, а не хранить вечную ссылку на первый render.

### Риск: хрупкие template-тесты

Снижение риска:

- делать акцент на проверку helper-результата и структуры ответа, а не на слишком жёсткие snapshot-проверки всей разметки.

## Критерии готовности

- Главная больше не рендерит все номера страниц, когда количество страниц превышает компактный порог.
- `Показати ще` продолжает работать и корректно обновляет пагинацию.
- Номерные ссылки по-прежнему ведут на реальные URL страниц.
- Блок пагинации остаётся компактным на mobile и desktop.
- В реализации используется один серверный источник истины для расчёта окна пагинации.
