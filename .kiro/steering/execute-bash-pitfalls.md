# execute_bash quirks в этом workspace

Эти правила выведены из реальных багов сессии. Учитывай их КАЖДЫЙ
раз перед запуском shell-команд.

## Базовые наблюдения

1. **`cwd` параметра недостаточно.** execute_bash переиспользует ту же
   zsh-сессию между вызовами. Если в прошлой команде был `cd`, новая
   команда всё равно запустится в той директории, а параметр `cwd`
   будет проигнорирован. **Решение:** использовать абсолютные пути в
   самой команде (`git -C /Users/zainllw0w/TwoComms/site ...`,
   `python /abs/path/manage.py ...`). НЕ полагаться на `cwd`.

2. **`Exit Code: -1`** в выводе НЕ означает, что команда упала.
   PTY не может корректно отдать код выхода в этой сессии. Реальный
   код проверяй через явный `; echo "RC=$?"` в конце команды или
   по содержимому stdout/stderr.

3. **Странные дубли букв** в выводе (`ssshpass`, `ttouch`,
   `cd "/Users/.../si tte/twocomms"`) — это zsh с подсветкой
   синтаксиса рендерит команду посимвольно через ANSI-эскейпы,
   они попадают в захваченный stdout. На реально исполняемую
   команду это не влияет — bash получает её правильно.

4. **`!` внутри двойных кавычек** включает history expansion в
   bash/zsh. Всегда **одинарные кавычки** для строк с `!` (пароли,
   паттерны и т.п.). Уже знакомое правило, но именно тут оно
   критично.

## Рекомендации

- **Длинные shell-конвейеры** — выноси в `.sh` файл в `tmp/`
  (workspace разрешает запись только туда), запускай через
  `bash /Users/zainllw0w/TwoComms/site/tmp/script.sh` с
  абсолютным путём, потом удаляй файл.
- **SSH-команды** на прод — пиши команду в одинарных кавычках,
  без вложенных bash-loginов (`ssh ... 'cmd1 && cmd2'` короче и
  устойчивее, чем `ssh ... "bash -lc '...'"`).
- **Длинные секреты в команде** — обязательно одинарные кавычки.
  Никогда не сохраняй пароль в файл (см. workflow.md).
- Если нужен реальный код возврата — добавь
  `... ; echo "RC=$?"` в конец, и парси по `RC=`.
- При первом запуске проверяй вывод глазами на наличие двойных
  букв/экранирования — это сигнал, что zsh пыталась дополнить
  команду. Если так — переписывай в скрипт.

## Проверочный sanity-check после ssh-команд

После каждой деплойной команды бери результат **из тела ответа**,
а не из `Exit Code`. Например:

```
ssh ... 'cd /path && git pull && echo PULL_OK'
```

Если в stdout есть `PULL_OK` — команда выполнилась. Если нет —
что-то пошло не так до этой строки.
