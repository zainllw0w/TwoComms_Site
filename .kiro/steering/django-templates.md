# Django-шаблоны: комментарии (профилактика баг-лика)

## Правило

- Многострочные комментарии — ТОЛЬКО `{% comment %}…{% endcomment %}`.
- `{# … #}` — ТОЛЬКО однострочно: открытие `{#` и закрытие `#}` на одной
  физической строке.

## Почему это критично

Лексер шаблонов Django использует `tag_re = ({%.*?%}|{{.*?}}|{#.*?#})`,
скомпилированный БЕЗ флага `re.DOTALL`. Поэтому `{#.*?#}` не сопоставляется
через перевод строки: многострочный `{# … #}` НЕ распознаётся как комментарий
и рендерится как ВИДИМЫЙ ТЕКСТ на странице, а HTML-теги внутри него браузер
ещё и выполняет.

Проверено на Django 5.2.11:

- многострочный `{# a⏎ b #}` → НЕ матчится (течёт текстом);
- однострочный `{# a #}` → матчится (нормальный комментарий).

Реальный инцидент: `management/_audio_player.html` начинался с 4-строчного
`{# … #}` и протекал текстом внизу каждой management-страницы (include в
`base.html`).

## Как правильно

```django
{% comment %}
Любой многострочный комментарий, включая <div>примеры</div>
и {{ шаблонный_синтаксис }} — всё игнорируется парсером.
{% endcomment %}
```

## Чего избегать

```django
{# Это многострочный комментарий,
   который ПРОТЕЧЁТ как текст на странице #}   ← БАГ
```

## Быстрая проверка перед коммитом шаблона

Грепнуть строки, где открыт `{#`, но нет `#}` на той же строке:

```
rg -n '\{#(?:(?!#\}).)*$' --glob '**/templates/**/*.html'
```

Любой результат, кроме намеренного однострочника, — потенциальный лик.
