2. Текущий этап проекта (факт)
2.1 Общая стадия


===== PAGE 2 =====
Проект не находится в стадии прототипа. Это рабочий DTF-поддомен с реализацией основных бизнес-
сценариев:
- лендинг/каталог/цены/требования/качество/блог
- заказ и lead-flow
- constructor MVP с сохранением сессии
- preflight-анализ файлов
- sample flow
- cabinet routes
- host-aware robots/sitemap
Подтверждение:
- `specs/dtf-codex/CHECKLIST.md` фиксирует исполненные Part 3/Part 4 блоки
- `specs/dtf-codex/EVIDENCE.md` фиксирует тесты, deploy, smoke
- `twocomms/dtf/tests.py` содержит расширенный набор сценариев (в т.ч. Part4/Part5/Part6)
2.2 Архитектурное состояние frontend
Фактический подход: Django SSR + progressive enhancement через JS.
Ключевые элементы:
- эффект-реестр и HTMX lifecycle:
  - `twocomms/dtf/static/dtf/js/components/core.js`
  - поддержка `htmx:afterSwap` и `htmx:beforeCleanupElement`
- утилиты эффекторного слоя:
  - `twocomms/dtf/static/dtf/js/components/_utils.js`
- бизнес/страничная логика:
  - `twocomms/dtf/static/dtf/js/dtf.js`
В `base.html` подключен модульный effect-layer (CSS+JS), но одновременно есть legacy-pattern 
(часть эффектов в `effect.*`, часть в именах без `effect.`).
2.3 Архитектурное состояние backend
- Валидация и preflight реализованы на backend:
  - `twocomms/dtf/forms.py` + `twocomms/dtf/preflight/engine.py`
- Preflight реально проверяет:
  - magic bytes, extension
  - размер
  - DPI
  - alpha/margins
  - color mode
  - PDF single-page/media box
- Constructor flow сохраняет preflight report и preview assets.
2.4 Что это значит по зрелости
Стадия: «рабочая продуктовая база + незавершенная унификация UI-effect слоя».
То есть:
- продукт уже можно развивать итеративно
- но перед масштабным добавлением новых эффектов нужно закрыть слой согласованности (единые 
контракты компонентов, cleanup legacy, SEO-safe правила, production prerequisites)

`Ideas/Audit/opus 4.6 v1.MD` и `v2.MD`

6. Критические ограничения и предпосылки перед следующей итерацией
6.1 Инфра/фоновые задачи
Факт по production:
- MySQL используется
- Celery settings присутствуют
- но фактическая связка Celery+Redis сейчас не операбельна (нет worker + broker DNS error)
Следствие:
- любые планы «реального polling статусов Celery task» сейчас ненадежны
- нужно сначала решить инфраструктурный prerequisite, либо формально принять sync/fake-step 
стратегию
6.2 Техдолг согласованности фронта


===== PAGE 6 =====
- В `dtf.js` остаются legacy-функции эффектов (минимум `initCompare`, `initTracingBeam`), не 
вызываемые в bootstrap
- В `js/components` и `css/components` есть дубли `effect.*` и старых версий
Это усиливает риск того, что разные агенты будут развивать разные версии одного и того же 
эффекта.
6.3 UX-конфликт правого нижнего угла
В `base.html` одновременно присутствуют:
- `#dtf-fab`
- `nav[data-floating-dock]`
- `nav.mobile-dock`
Это давно зафиксированная проблема, подтвержденная несколькими отчетами.

7. Что интегрировать в проект в первую очередь (без обсуждения сложности)
Здесь фокус только на ценности и правильности интеграции, как вы просили.
1. Единый контракт «Quick Actions Layer»
- Утвердить поведение `FAB/Dock/Mobile Dock` как одну систему, а не три конкурирующих блока.
- Определить single source of truth по breakpoint-логике и z-index.
2. Multi-Step Loader как честный state-machine
- Либо реально привязка к backend step-status
- Либо прозрачная синхронная модель без имитации «несуществующих шагов Celery».
3. Compare как доверительный коммерческий блок
- Оставить как ключевой доказательный компонент (до/после, 72 vs 300 DPI, RGB vs print 
reality).
- Добавить сценарии stop autoplay on interaction + mobile-first gesture logic.
4. Vanish Input до бизнес-контракта
- clear-on-invalid
- optional placeholder cycling
- не применять indiscriminately ко всем полям
5. Infinite Cards в SEO-safe формате
- first track = реальный HTML контент
- clone track = `aria-hidden`
- корректный fallback без JS
6. Cleanup и унификация effect-layer
- вынести Flip Words из `dtf.js` в `effect.flip-words.js`
- удалить/архивировать неиспользуемые legacy-дубли
- привести именование к единому стандарту
