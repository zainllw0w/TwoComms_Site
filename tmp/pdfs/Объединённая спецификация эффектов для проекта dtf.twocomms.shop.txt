Compare (До/После изображений)
Название и назначение эффекта: Эффект сравнения двух изображений «до/после» с
помощью движимого ползунка. Позволяет пользователю наглядно сравнить исходное
изображение с результатом обработки, имитируя референсный компонент Aceternity.
Поддерживает режимы взаимодействия drag (перетаскивание ползунка), hover
(автоматическое перемещение при наведении курсора на десктопе) и режим autoplay
для автоматического показа изменений. При любом действии пользователя
автоматическое воспроизведение должно останавливаться. На мобильных
устройствах, где эффект наведения недоступен, используется упрощённый режим
(например, перетаскивание вручную).
Техническая реализация: Эффект должен быть реализован без использования React-
рантайма, с опорой на существующий стек Django SSR + HTMX + Vanilla JS. Изначально
HTML-шаблон отрисовывает две изображения и полосу разделителя (ползунок) на сервере
(SSR), чтобы контент был доступен поисковым системам и без JS. JavaScript-код (например, 
effect.compare.js) при загрузке страницы инициализирует интерактивность:
добавляет обработчики событий для перетаскивания ползунка (mouse/touch events) и для
режима наведения (слушает движение курсора) и запуска/остановки автопроигрывания.
Для плавной анимации можно использовать CSS-трансформации и transition, либо
легковесную анимацию через requestAnimationFrame. Особое внимание – адаптация к 
prefers-reduced-motion: если пользователь предпочитает уменьшенную анимацию,
автопрокрутка не запускается. Реализация должна минимизировать внешние
зависимости и монтироваться на существующий lifecycle эффектов (через core.js или
аналогичный реестр).
Требования по SEO, SSR, доступности и i18n: Оба изображения и текстовые
альтернативы (атрибуты alt) должны присутствовать в HTML для индексирования и
доступности. Эффект не должен вызывать layout shift – контейнер имеет фиксированный
размер, и наложение ползунка не смещает макет. Для доступности ползунок ведёт себя
как элемент ARIA slider: фокусируемый, с возможностью управления с клавиатуры
(напр. стрелками влево/вправо для изменения положения). Необходимо обеспечить
соответствующие ARIA-атрибуты (ролевая роль слайдера, метки для значений до/после). В
режиме autoplay должен быть атрибут data-autoplay="true" (или false при
отключении) и data-compare-mode указывающий текущий режим (drag|hover|
autoplay) . Все видимые текстовые подсказки (если таковые есть) должны
поддерживать перевод (i18n), хотя сам эффект сравнения по сути языконезависим.
Архитектура подключения и примеры интеграции: В проекте предусмотрены
отдельные статические файлы для логики и стилей эффекта. JavaScript-код располагается в 
twocomms/dtf/static/dtf/js/components/effect.compare.js, стили – в twocomms/
dtf/static/dtf/css/components/effect.compare.css. Эти файлы подключаются в
базовом шаблоне или на нужных страницах. HTML-разметка эффекта присутствует,
например, в шаблонах главной или галереи: 
twocomms/dtf/templates/dtf/index.html, gallery.html, а также в

и a11y-требований; откладывание этих деталей ухудшит UX для части аудитории

Multi-step Loader + Preflight (Многошаговый загрузчик +
проверка файла)
Название и назначение эффекта: Многошаговый индикатор загрузки/проверки,
используемый при загрузке файлов (preflight check). Цель – пошагово отобразить процесс
проверки загруженного файла, заменив текущую фейковую задержку реальными этапами
проверки . Эффект должен последовательно показать ключевые шаги: (1) файл
получен, (2) проверка формата и сигнатуры, (3) DPI и физические размеры, (4)
прозрачность и границы, (5) риск тонких линий/деталей, (6) итоговый результат и
рекомендуемые действия. Пользователь получает понятный фидбек по каждому этапу
(PASS/WARN/FAIL) с пояснениями и дальнейшими шагами, чтобы улучшить UX доверия к
загрузке.
Техническая реализация: Реализация эффекта должна сочетаться с backend-логикой без
использования Celery/Huey на данном этапе (т.к. инфраструктура воркера нерабочая) –
то есть sync-first подход . При отправке файла форма обрабатывается синхронно
во вьюхе Django, которая выполняет все проверки (в модуле preflight/engine.py)
последовательно и быстро. Frontend-эффект может запускаться сразу при начале загрузки
(например, по событию формы через HTMX или JS) и показывать шаги. Важно: эффект не
должен искусственно задерживать шаги – если все проверки завершились практически
мгновенно, интерфейс должен либо быстро проскочить через этапы, либо сразу
отобразить завершение, не симулируя несуществующие задержки. Для реализации
этого можно либо:
Обновлять UI по мере готовности каждого шага (потребуется либо стриминг/Server-Sent
Events, либо периодический AJAX-запрос на состояние). Например, сделать отдельный API

проверки и её статус (галочка, предупреждение или крестик). По завершении всех шагов модуль
either показывает сводку или автоматически закрывается, предоставляя итоговые результаты. -
Требования по SEO, SSR, доступности и i18n: Данный эффект является вспомогательным UI, он
не  влияет  на  индексируемый  контент  страницы  (результаты  проверок  могут  содержать  текст
ошибок – они должны быть выведены в DOM, чтобы пользователь их видел, но эти сообщения
могут не требоваться индексировать). SSR здесь минимально важен: желательно, чтобы базовая
структура  многошагового  лоадера  (контейнер,  список  шагов)  присутствовала  в  HTML  при
рендеринге страницы для доступности, либо могла быть вставлена через HTMX как фрагмент. Для
доступности необходимо: каждый шаг снабдить ясным текстовым описанием (например, «Шаг 3:
DPI проверен, OK»), чтобы screen reader мог это прочитать. Можно использовать роль status
или  progressbar для контейнера, обновлять  aria-live региона при смене шагов, чтобы
озвучивать  изменения.  По  окончании  проверки  обеспечить  фокус  на  результате  или  кнопку
закрытия, чтобы не зациклить пользователя. i18n: все тексты шагов («Файл принят», «Проверка
DPI»  и  пр.)  должны  браться  из  словаря  перевода,  а  не  быть  захардкожены  на  одном  языке.
Например, можно хранить шаблоны сообщений для PASS/WARN/FAIL по коду проверки. Также
стоит учитывать, что числительные и форматы могут отличаться в разных языках. - Архитектура
подключения  и  примеры  интеграции: Эффект  интегрируется  в  формы  загрузки  файлов  на
страницах Order и Constructor, поэтому изменения нужны минимум в шаблонах twocomms/dtf/

запускаться по событию отправки файла. Возможно использование HTMX: например, атрибут
hx-target на  форму,  чтобы  предотвратить  полный  перезагруз  страницы  и  вместо  этого
получить от сервера промежуточный отклик, активирующий лоадер. Backend-часть: twocomms/
dtf/preflight/engine.py уже содержит бизнес-логику проверок – фронтенду нужно лишь
отображать  результаты.  При  необходимости  можно  добавить  URL  (в  urls.py)  и  view
( views.py) для AJAX получения статусов, если выбрана модель опроса сервера.  Пример:
Пользователь  нажимает  "Загрузить  файл",  сразу  выполняется  JS,  показывающий  overlay  с
шагами. Шаг 1 «Файл принят» отображается, параллельно начинается отправка файла. После
получения ответа или по таймерам, быстро проходят шаги 2-5. Шаг 6 отображает «Итог: найдено
X предупреждений» или «Все проверки пройдены». Затем предоставляется ссылка или действие
(например  «Продолжить»  либо  описание  ошибок).  В  случае,  если  проверка  провалилась  на
середине (например, недопустимый формат), можно сразу перейти к итогу с сообщением FAIL и
остановить  дальнейшие  шаги.  -  Технические  долги  и  потенциальные  риски: В  текущей
реализации эффект  фейковый – шаги переключаются по фиксированным таймерам и не
отражают реальный ход проверки. Это создает риск рассинхронизации: если проверка займёт
меньше  или  больше  времени,  интерфейс  всё  равно  отрабатывает  по  шаблону,  вводя  в
заблуждение пользователя. Главный долг – связать UI с фактическими результатами engine.py.

Vanish Input (Исчезающее поле ввода)
Название и назначение эффекта: Эффект интерактивного поля ввода, которое особым
образом реагирует на некорректный ввод. Цель – улучшить UX валидции форм: при
неверном вводе поле не просто подсвечивается красным, а стряхивается (shake) и
“исчезает” – т.е. содержимое плавно растворяется и очищается, сбрасывая неверные
данные . Дополнительно, placeholder (текст-заполнитель) поля циклически меняется,
подсказывая пользователю формат или примеры ожидаемого ввода. Таким образом,
эффект одновременно привлекает внимание к ошибке и дает подсказки, что повысит шанс
корректного заполнения со второй попытки.
Техническая реализация: Эффект реализуется на чистом JS/CSS поверх стандартного 
<input> (или <textarea>) элемента, без сторонних фреймворков. В HTML-шаблоне
поле ввода содержит начальный placeholder (например, первый из списка подсказок). При
подключении скрипта vanish-input.js, он расширяет функциональность поля: 
На событие invalid (HTML5 валидация) или на пользовательское событие проверки
(например, потеря фокуса с неправильным значением) скрипт добавляет классы для
анимации: тряска (shake) и затем плавное исчезновение текста. Можно достичь этого
через CSS: например, анимация shake (несколько смещений на 0.1s), затем плавная смена
opacity текста до 0. После окончания анимации JS очищает value поля.
Placeholder cycling: реализуется таймером или при каждом фокусе. Скрипт хранит массив
строк (подсказок), переданный через data-атрибут или глобально. Через заданный
интервал он меняет input.placeholder на следующий вариант. Например, каждые 3
секунды прокручиваются разные вопросы/фразы. При фокусе пользователя на поле можно
приостановить смену, чтобы не отвлекать.
При отправке формы (с valid значением) эффект не вмешивается: поле ведёт себя обычно,
без исчезновения содержимого.
Для мобильных устройств и слабых устройств предусмотрено упрощение: без тяжёлого canvas и
сложных эффектов. Например, если планировалась очень сложная анимация растворения
через canvas-пикселизацию, на мобильном можно ограничиться простым fade-out. Определение
мобильного  может  быть  через  CSS  media  query  (pointer:coarse)  или  user-agent.  В  целом,
предпочтительно использовать CSS-анимации (транзиции opacity, transform) вместо canvas, т.к.

поле  пропадёт  из  фокуса).  Кроме  того,  сменяющиеся  placeholder’ы  не  должны  быть
единственным источником подсказки по смыслу (поскольку скринридеры их не озвучивают, если
поле уже в фокусе). Стоит продублировать подсказку в виде скрытого пояснения (ARIA description)
или статического текста рядом с полем. i18n: набор placeholder-строк должен быть локализован.
Например, в английской версии: ["Enter your email", "Example: user@example.com"], в украинской
версии – свои примеры. Лучше получать этот список с сервера (через контекст шаблона или JSON
в  data-атрибуте)  на  основе  текущей  локали.  -  Архитектура  подключения  и  примеры
интеграции: Скрипт  vanish-input.js подключается  в  общую  сборку  или  на  страницы  с
соответствующими  формами.  Стили  (vanish-input.css)  определяют  ключевые  анимации
(shake, fade-out) и классы. Интеграция минимально требует добавить атрибуты к нужным полям,
чтобы  скрипт  их  инициализировал  –  например,  data-effect="vanish-input" и,
опционально,  data-placeholders='[...]' с  JSON-массивом  подсказок.  Пример
использования:  поле  ввода  для  промокода  на  странице  оформления  заказа.  В  order.html
вокруг  <input name="promo_code"> можно добавить span-обёртку для визуальных эффектов
или  иконки  ошибки.  После  подключения  скрипта  неверный  промокод  будет  вызывать
анимацию: поле слегка потрясётся и очистится. В текущем проекте явных упоминаний шаблонов
в спецификации нет, но, скорее всего, эффект применим к полям, где ожидается определённый
формат (конструктор, поиск или форма заказа). Нужно убедиться, что во всех таких местах input’ы
помечены для подключения эффекта. Файлы интеграции: скрипт twocomms/dtf/static/dtf/
js/components/vanish-input.js и  стили  twocomms/dtf/static/dtf/css/components/
vanish-input.css. -  Технические долги и потенциальные риски: Текущая реализация
находится  на  уровне  минимума –  вероятно,  сейчас  поле  лишь  пульсирует/тряётся  при
ошибке,  но  не  реализованы  эффекты  растворения  и  очистки,  также  placeholders  могут  не
циклироваться. Этот долг следует устранить, иначе теряется смысл названия "Vanish". Риск
при  реализации  –  перегруженность  эффекта:  слишком  частая  смена  placeholder’ов  может
отвлекать,  поэтому  следует  подобрать  умеренный  интервал  и  останавливать  цикл,  когда
пользователь  начинает  ввод.  Ещё  риск  –  конфликт  с  браузерной  автозаполнением:  если
браузер  вставит  значение  автоматически,  нужна  логика,  чтобы  не  считать  это  ошибкой.
Технический долг в коде: убедиться, что эффект не ломает форму (например, после исчезновения
текста форма всё ещё сабмитится нормально). Также важно учитывать  mobile experience: на
мобильных устройствах поле в фокусе вызывает экранную клавиатуру – анимации в этот момент
могут  мешать.  Потенциально,  для  input[type=email] или  других  специфических  полей
placeholders  должны  соответствовать  формату  (например,  показывать  user@domain.com  для
email). Наконец, нужно проверить, что при отключенном JavaScript базовая валидация всё равно
доступна  (HTML5-атрибуты  required/pattern  работают).  -  Пути  к  шаблонам  и  ресурсам:

dtf/static/dtf/css/components/vanish-input.css –  стили  анимации .  Шаблоны

Floating Dock и навигация (нижняя панель + Sidebar
конструктора)
Название и назначение эффекта:Floating Dock – плавающая док-панель à la macOS
Dock, используемая как основная навигация на сайте. Назначение – объединить разные
элементы навигации в единый понятный компонент. В текущем интерфейсе
присутствовал конфликт: одновременно использовались кнопка-действие в правом
нижнем углу (FAB), мобильная навигационная панель и экспериментальный floating dock,
что создавало конкуренцию между тремя элементами. Новая цель: оставить один
главный паттерн навигации – плавающий dock внизу для десктопа (с эффектом
масштабирования иконок при наведении), а для мобильных – аналогичный нижний бар
без эффекта hover-увеличения. Дополнительно, для страницы конструктора (где
пользователь, вероятно, редактирует или загружает что-то) – предусмотреть левый
боковой тулбар (sidebar), содержащий инструменты, отдельный от основной навигации
. Таким образом, Floating Dock обеспечивает глобальную навигацию, а sidebar –
контекстные действия конструктора.
Техническая реализация: Floating Dock состоит из набора ссылок-иконок, отображаемых
горизонтально (навигационные пункты). На десктопе применяется эффект приближения:
иконки возле курсора увеличиваются плавно, имитируя 3D-приближение. Реализовать это
можно с помощью отслеживания положения мыши: при движении курсора над областью
Dock вычислять расстояние до центра каждой иконки и задавать ей scale в зависимости от
близости (например, по гауссовой кривой). В библиотеке Aceternity UI для этого
используется MotionValue и возможно Framer Motion, но в нашем случае лучше
применить чистый JS (например, requestAnimationFrame с вычислением transform:
scale). Для мобильной версии (touch) эффект увеличения отключается – вместо него иконки
могут просто подсвечиваться при тапе. Поскольку dock всегда виден внизу, нужно учесть
адаптивность: на мобильных он фиксирован снизу экрана (можно использовать CSS 
position: fixed с bottom: 0), а на десктопе – тоже фиксирован, либо абсолютный
внутри layout, но всегда внизу видимой области. Sidebar (левая панель конструктора) – это
вертикальное меню, появляющееся только на странице конструктора. Его реализация
проще: при загрузке страницы конструктора скрипт может инициализировать раскрытие/
свертывание sidebar (если нужно, например, скрывать на маленьких экранах), либо просто
стилизовать statically. Sidebar содержит, например, кнопку «Загрузить изображение»,
«Настройки» и пр. – фактически инструменты, которые не должны дублироваться в
основном Dock. Их можно оформить как список значков с подсказками (tooltip) при
наведении. Так как Sidebar существует только на одной странице, можно не делать его
суперуниверсальным компонентом, а реализовать напрямую в шаблоне конструктора с
минимальным JS (например, для tooltip или для анимации появления).
Требования по SEO, SSR, доступности и i18n: Навигационные элементы должны быть
представлены стандартными HTML элементами: лучше всего <nav> контейнер, внутри 
<a> ссылки с значками и, возможно, видимыми или скрытыми текстовыми метками. Это
обеспечит, что поисковые системы видят структуру сайта (ссылки на основные разделы) –
это важно для SEO. SSR: Dock и Sidebar отрисовываются на сервере (в base.html для Dock
и в constructor_app.html для sidebar) так, чтобы при отключенном JS они всё равно
были функциональны (пусть и без эффекта увеличения). Доступность: каждая иконка
должна иметь текстовое описание. Это можно сделать либо видимым (например, подпись
под иконкой или всплывающая при наведении), либо скрытым (aria-label).
Пользователь с клавиатурой должен иметь доступ к навигации: ссылки должны быть в
DOM-порядке для табуляции. Если Dock позиционируется фиксированно, желательно
добавлять его в конец DOM, либо управлять tabindex, чтобы порядок фокуса был

Infinite Moving Cards (бесконечные движущиеся карточки
для блога)
Название и назначение эффекта: Эффект бесконечно движущейся ленты карточек,
используемый для раздела статей блога/базы знаний на главной странице. Цель –
привлечь внимание к нескольким постам за счёт анимированного скролла, но при этом
сохранить SEO-видимость контента. Основной контент (карточки статей с
заголовками/изображениями) должен быть индексируемым, а движущаяся копия – сугубо
декоративной. Иными словами, реальный HTML карточек присутствует для поисковиков, а
анимация достигается с помощью дублирования этих элементов и непрерывного их
прокручивания.
Техническая реализация: Карточки формируются на бэкенде: например, берутся
последние N статей блога и выводятся как список <article> или <div> элементов. Для
эффекта бесконечного движения делается клон этого списка: в верстке можно либо
дважды продублировать блок (второй – для прокрутки), либо сгенерировать один, а
второй создавать/клонировать динамически через JS. Проще реализовать на стороне
клиента: иметь одну копию в HTML (индексируемую), а при загрузке страницы скрипт 
effect.infinite-cards.js клонирует ее содержимое, помечает атрибутами для
исключения из доступа и запускает анимацию прокручивания. Анимация: можно
использовать CSS @keyframes – например, перемещать ленту влево на 100% за
определённое время и зациклить (в Effects.MD показан подход с keyframes scroll и
custom property для скорости). Альтернативно, можно использовать JS + CSS: по 
requestAnimationFrame смещать позицию ленты. CSS предпочтителен, т.к. позволяет
плавный непрерывный скролл без нагрузки на основной поток. Pause on hover/focus:
обеспечить, что при наведении курсора на блок или при фокусе на элементе анимация
останавливается. В CSS это можно реализовать через :hover изменяющий 
animation-play-state: paused для контейнера, а JS – добавить/убрать класс paused.
Также при фокусе (через JS: onfocus каждого элемента) ставим анимацию на паузу, чтобы
пользователь мог прочитать/кликнуть. Reduced-motion fallback: через медиа-query 
prefers-reduced-motion: reduce – в этом случае вообще не запускаем анимацию, а
оставляем блок статичным или со скроллом вручную. Например, можно ограничиться
тем, что пользователь может пролистать карточки вручную (горизонтальный скролл), либо
отобразить их как статическую сетку.

Speed/Tremble Text Effect (эффект скорости/дрожания текста)
Название и назначение эффекта: Небольшой текстовый эффект для подчеркнутых фраз,
создающий ощущение вибрации или рывка, ассоциирующегося со скоростью. Цель –
акцентировать внимание на ключевом маркетинговом сообщении (например, фраза
«відправка день у день» – "отправка в тот же день") с помощью анимации, которая
подсознательно воспринимается как скорость. Эффект короткий по времени и
применяется точечно к одному-двум словам или короткой фразе, чтобы не отвлекать
надолго.
Техническая реализация: Данный эффект лучше всего реализовать при помощи CSS-
анимации или минимального JS. Один из вариантов – дрожание текста: быстрый сдвиг
букв на пару пикселей влево-вправо и возвращение, либо размытие (motion blur) на долю
секунды. Например, можно задать ключевые кадры @keyframes tremble с
трансформацией translateX(-1px) и translateX(1px) в чередующихся шагах,
создавая вибрацию, затем возвращение к норме. Длительность всей анимации – короткая,
порядка 0.5-1 секунды (критерий – краткая длительность). Анимация запускается
один раз при загрузке страницы или при появлении элемента в области видимости. Чтобы
гарантировать, что текст остаётся читаемым, эффект не должен быть затяжным или
чрезмерным – никаких постоянных циклов. Prefers-reduced-motion: если пользователь
против анимаций, эффект не запускается (можно обернуть с помощью @media (prefers-
reduced-motion: reduce) и просто не применять keyframes или задать длительность 0)
. Реализация на сервере: добавить CSS-класс к нужному элементу, например <span 
class="speed-effect">текст</span>. В CSS определить для .speed-effect

0.7s ease-out 1), или добавить этот класс с JS после небольшой задержки для
синхронизации с появлением элемента. Можно также использовать JS IntersectionObserver ,
чтобы запустить анимацию, когда элемент появился в зоне видимости (полезно, если текст

Требования по SEO, SSR, доступности и i18n:SEO: Так как эффект применяется к

понадобится слой для размытия, то с помощью CSS (например, текст-тень) либо SVG-
фильтра, но не через копирование строки, иначе поисковик увидит повтор. SSR: текст
изначально присутствует на странице без эффектов, чтобы при отключённом JS/CSS
пользователь всё равно увидел обычный текст (просто без анимации). Доступность:
анимация достаточно микроскопическая, но всё же – не должна вызывать дискомфорт.

эффект для тех, кому он противопоказан. Также, убедиться, что эффект не ломает
чтение: например, скринридер прочитает фразу нормально, потому что в DOM она одна. 
i18n: сам текст, конечно, должен переводиться как обычно, эффект на любой языковой
версии идентичен. Нужно проверить, не потеряется ли смысл в других языках – например,
если фраза длиннее, эффект всё равно должен быть кратким. При необходимости, можно
применять его только к части фразы или регулировать длительность.

Speed/Tremble Text:
Как определить, где применять этот эффект? Будет ли он размечен вручную в шаблонах
(как класс speed-effect для конкретных слов) или возможно, что маркетинг/редакторы
смогут как-то помечать нужные места через CMS? Здесь вопрос к архитектуре: centralize ли
логику, например, в виде шаблонного фильтра или реактивного компонента, который
оборачивает текст, или довериться ручной вставке. 
Насколько гибким задуман эффект – только ли один вид (горизонтальное дрожание), или
нужно предусмотреть вариации (например, легкое размытие vs дрожание)? Если
вариации, то как переключать – доп.классами (speed-effect--blur)? Этот вопрос
предвосхищает потенциальное расширение функциональности и поддержки кода.
