# 14 — Git + Deploy workflow (обязательно соблюдать)

## Контекст
Проект деплоится на сервер через SSH. Мы не “ведём локальную сборку руками” — делаем изменения, коммитим, пушим и выкатываем.

---

## 1) Правило больших изменений
После каждого крупного блока (хедер, hero, форма зразка, требования, калькулятор):
1. Проверить локально/на превью как принято в проекте (build/lint/tests).
2. `git status` / `git diff`
3. **commit** (желательно понятный conventional‑style):
   - `feat(header): ...`
   - `fix(i18n): ...`
   - `refactor(cta): ...`
4. **push** в основной репозиторий.
5. Деплой через SSH.

---

## 2) SSH деплой (не угадывать!)
CODEX должен:
- найти в репозитории существующую документацию деплоя (README / ops / deploy scripts),
- использовать ровно тот способ, который уже принят.

Если документации нет:
- создать `DEPLOY_NOTES.md` и описать, что удалось найти на сервере:
  - путь проекта,
  - сервис (docker/systemd/pm2),
  - команды обновления,
  - логи.

---

## 3) Минимальный чек деплоя
После выката:
- открыть главную и 2–3 ключевые страницы
- проверить хедер (все языки)
- проверить CTA
- проверить форму “тест‑пак”
- проверить калькулятор
- проверить, что нет debug строк

---

## 4) Ведение истории
- В `CHANGELOG.md` (по шаблону) добавить:
  - список коммитов
  - кратко, что изменилось
  - что проверить вручную
