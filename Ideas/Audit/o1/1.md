### 1.2 Реалистичность технических решений

| Решение | Реалистичность на cPanel shared | Комментарий |
|---|---|---|
| Redis/Celery fix | ❌ Нереалистично | Shared hosting не даёт произвольных демонов. Даже если установить Redis — процесс будет убит cPanel при OOM. |
| Huey (SQLite broker) | ⚠️ Возможно с оговорками | Huey с SQLite работает, но нужен постоянный worker процесс. На cPanel — через `nohup` или cron-based polling. Ненадёжно. |
| Sync-only preflight | ✅ Реалистично | Preflight проверки (DPI, magic bytes, size) занимают <2 сек для файлов до 50MB. Sync вполне допустим. |
| HTMX polling | ✅ Реалистично | `hx-trigger="every 2s"` для status check. Работает на любом хостинге. |
| CSS-only animations | ✅ Оптимально | Zero JS overhead, GPU-accelerated, battery friendly. |
| Canvas/WebGL | ⚠️ Только лёгкие | Допустимо для одного элемента на странице. Не для backgrounds. |

## 2. Market/UX Research

### 2.1 Как DTF/Print-on-Demand сервисы показывают Preflight

**Printful** (printful.com):
- Загрузка файла → мгновенная проверка DPI (синхронная) → preview на mockup
- Ошибки показаны inline под полем загрузки, не в modal
- Тон: дружелюбный, не технический. "Your image resolution is low. For best results, use 300 DPI."
- Нет multi-step loader — один запрос, один ответ
- **Вывод:** Sync preflight с inline feedback — индустриальный стандарт

**Printify** (printify.com):
- Drag-and-drop upload area
- Проверка на стороне клиента (JS): размер файла, формат
- Серверная проверка: DPI, dimensions → результат в течение 1-3 сек
- При ошибке: жёлтый banner сверху, dismiss-кнопка
- **Вывод:** Client-side pre-validation снижает нагрузку на сервер

**CustomInk** (customink.com):
- Многошаговый wizard: Design → Review → Approve
- Preflight визуально встроен в "Review" шаг, не отдельный процесс
- Если DPI низкий: overlay на preview с размытием + текст "Image may appear blurry when printed"
- **Вывод:** Показывать проблему визуально на preview — мощнее текстовых ошибок

**SPOD/Spreadshirt**:
- Автоматический upsample если DPI < 150 (с предупреждением)
- Progress bar только для больших файлов (>10MB)
- **Вывод:** Для файлов <10MB progress bar не нужен — instant feedback

**Общий паттерн индустрии:**
1. Синхронная проверка (не async!)
2. Inline feedback (не модальные окна)
3. Визуальная демонстрация проблемы (размытый preview)
4. Дружелюбный copy без технических терминов
5. Возможность продолжить с предупреждением (для WARN)

### 2.2 Best Practices для Before/After Comparisons

**Исследования и источники:**

- **Nielsen Norman Group** (nngroup.com) — "Comparison UX": split-screen comparisons наиболее эффективны когда пользователь контролирует разделительную линию. Autoplay снижает perceived control.
- **Baymard Institute** — image comparison в e-commerce: hover-to-reveal предпочтительнее drag на mobile (палец закрывает контент при drag).
- **Apple.com** (product comparisons) — использует scroll-triggered reveal: при скролле изображение плавно переключается. Никакого ручного drag.

**Рекомендации для DTF:**

| Режим | Desktop | Mobile | Когда использовать |
|---|---|---|---|
| Drag slider | ✅ Primary | ⚠️ Secondary | Детальное сравнение (Gallery) |
| Hover reveal | ✅ Good | ❌ Impossible | Quick preview (Landing hero) |
| Autoplay loop | ⚠️ Attention grabber | ✅ Good | Testimonials, первое знакомство |
| Tap toggle | ❌ Unintuitive | ✅ Primary | Mobile-first contexts |

**Ключевой инсайт:** Autoplay → привлекает внимание, но пользователь не чувствует контроля. Оптимально: **autoplay при входе в viewport → пауза → ожидание interaction.**

### 2.3 Motion Patterns для E-commerce конверсии

**Исследования:**

- **Google Material Design Guidelines** — motion должен быть functional, не decorative. Каждая анимация должна либо давать feedback, либо направлять внимание, либо показывать связь между элементами.
- **Shopify UX Research (2023)** — микроанимации на кнопках "Add to Cart" увеличивают click-through на 3-7%. Но анимации дольше 300ms снижают perceived performance.
- **Stripe.com** — benchmark для B2B интерфейсов: subtle, functional, fast. Анимации ≤200ms для UI feedback, ≤500ms для transitions.

**Применимость к DTF:**
- ✅ Stateful Button (loading/success/error) — прямой impact на конверсию
- ✅ Encrypted Text — привлекает внимание к заголовкам, уместно для tech-бренда
- ✅ Compare slider — демонстрирует ценность продукта
- ⚠️ Sparkles, BG Beams — декоративные, не влияют на конверсию напрямую
- ❌ Particle systems, Liquid animations — отвлекают от CTA

### 2.4 SEO для JS-heavy Animations

**Google Search Central Documentation:**

- Googlebot рендерит JavaScript, НО: execution budget ограничен (~5 секунд)
- CSS анимации НЕ влияют на индексацию
- JS, который модифицирует DOM (клонирование контента) — может создать duplicate content
- `aria-hidden="true"` корректно игнорируется Googlebot
- Core Web Vitals: CLS (Cumulative Layout Shift) — анимации, сдвигающие контент, наказываются

**Рекомендации:**
1. Все текстовые эффекты (Encrypted Text, Text Generate) — SSR контент должен быть в HTML до JS. JS только добавляет визуальный слой.
2. Infinite Cards — первый track = реальный HTML, клоны = `aria-hidden`.
3. Ни один эффект не должен вызывать layout shift. Использовать `transform` и `opacity` only.
4. `will-change` property для анимируемых элементов (но не более 3-4 на страницу).

## 3. Behavioral Model

### 3.1 Сегмент 1: Новички (60%)

**Профиль:**
- Первый заказ DTF-печати
- Пришли из поисковой выдачи или Instagram
- Не знают что такое DPI, CMYK, bleed margins
- Средний бюджет: 1-5 единиц
- Главный страх: "потрачу деньги и получу фигню"
- Решение о покупке: 3-7 дней

**Поведенческая модель:**

```
Landing → Понимание (что такое DTF?) → Trust-building (отзывы, примеры)
→ Цены (сколько стоит?) → Sample Kit (можно попробовать?)
→ Upload (боюсь ошибиться) → Preflight (что это за ошибки?!)
→ Заказ (надеюсь всё правильно)
```

**UX потребности:**
| Этап | Потребность | Эффект/Решение |
|---|---|---|
| Landing | Понять за 5 сек что такое DTF | Compare slider (autoplay, 72 vs 300 DPI) |
| Trust | Увидеть реальные результаты | Infinite Cards (отзывы), Gallery |
| Цены | Простой расчёт без таблиц | Калькулятор с instant preview |
| Upload | Не бояться ошибиться | Drag-and-drop с превью + подсказки |
| Preflight | Не паниковать при ошибках | Friendly copy + visual demo проблемы |
| Заказ | Уверенность в решении | Stateful Button + confirmation screen |

**Effects для новичков:**
- ✅ Compare (с подписями "так видит экран" / "так напечатается")
- ✅ Tooltip Card (пояснения терминов DPI, CMYK, bleed)
- ✅ Text Generate (trust-блоки: "Мы напечатали 50,000+ принтов")
- ✅ Progressive disclosure (развёрнутые пояснения по умолчанию)
- ⚠️ Encrypted Text (может сбить с толку — "что за мусор на экране?")

### 3.2 Сегмент 2: Профессионалы (40%)

**Профиль:**
- Регулярные заказы (2-4 раза в месяц)
- Знают требования к файлам
- Ценят скорость workflow: upload → check → pay → done
- Средний бюджет: 20-100+ единиц
- Главный страх: "медленно, неудобно, переключусь на другого"
- Решение: мгновенное (есть доверие)

**Поведенческая модель:**

```
Кабинет → Upload (пакетно) → Preflight (быстро, без лишнего)
→ Оплата → Отслеживание → Повтор заказа
```

**UX потребности:**
| Этап | Потребность | Эффект/Решение |
|---|---|---|
| Вход | Быстрый доступ к заказу | Floating Dock → "Новый заказ" |
| Upload | Batch upload, drag-and-drop | Минимум UI, максимум функции |
| Preflight | Только ошибки, без воды | Compact mode: иконка + текст |
| Оплата | One-click reorder | Saved payment methods |
| Кабинет | История, статусы, repeat | Таблица заказов с фильтрами |

**Effects для профи:**
- ✅ Stateful Button (instant feedback)
- ✅ Skeleton loading (perceived speed)
- ✅ Keyboard shortcuts (upload: Ctrl+U, submit: Ctrl+Enter)
- ❌ Compare (уже знают разницу — раздражает)
- ❌ Tooltip Card (знают терминологию)
- ❌ Autoplay anything (отвлекает)

### 3.3 Adaptive Complexity: Реализация

**Механизм:**

```
1. Первый визит → "Newcomer mode" (default)
   - Развёрнутые пояснения
   - Autoplay compare
   - Tooltip cards active
   - Preflight: verbose mode

2. После 1-го успешного заказа → предложить "Pro mode"
   - Свёрнутые пояснения
   - Compare = manual only
   - Tooltips = disabled
   - Preflight: compact mode

3. Хранение: localStorage key "dtf_ux_mode" = "newcomer" | "pro"

4. Toggle в footer/cabinet: "Упрощённый режим | Экспертный режим"
```

**Технически:**
- Django context: `{{ ux_mode }}` из cookie/session
- CSS: `.ux-newcomer .pro-only { display: none }` / `.ux-pro .newcomer-only { display: none }`
- JS effects: `if (DTF.uxMode === 'pro') return;` в init

## 4. Forecast & Scenarios (90 дней)

### Сценарий A: Conservative (только стабилизация)

**Scope (недели 1-12):**

| Неделя | Задача |
|---|---|
| 1-2 | Унификация effect lifecycle. Удаление legacy из dtf.js. Удаление дублей файлов. |
| 3-4 | Error boundary для всех эффектов. prefers-reduced-motion. Feature flags. |
| 5-6 | Fix Compare: добавить hover mode, keyboard a11y. |
| 7-8 | Fix Infinite Cards: бесшовный scroll, SEO-safe, pause on hover. |
| 9-10 | Multi-Step Loader → Sync preflight с inline feedback (без fake таймеров). |
| 11-12 | FAB/Dock конфликт: единый навигационный паттерн. QA pass. |

**Плюсы:**
- Минимальный риск регрессий
- Всё существующее начинает работать правильно
- Фундамент для будущих фич
- Техдолг сокращается на ~70%

**Риски:**
- Нет WOW-эффекта для пользователей
- Конкуренты могут уйти вперёд визуально
- Команда может потерять мотивацию (только "починки")

**Метрики контроля:**
- File upload success rate: текущий → >80% (за счёт sync preflight)
- JS errors в production: → 0 critical
- Core Web Vitals: all green
- Preflight pass rate: → >70% (за счёт guidance копирайта)

**Ожидаемый impact на конверсию:** +5-10% (за счёт устранения broken flows)

**Scope (недели 1-12):**

| Неделя | Задача |
|---|---|

**Плюсы:**
- Баланс между стабильностью и impression
- 3 заметных визуальных улучшения для пользователей
- Floating Dock → запоминаемость бренда
- Text Generate → storytelling для новичков

**Риски:**
- Средний риск: WOW-фичи могут потребовать больше времени
- Performance impact от 3 новых JS эффектов — нужен мониторинг
- Если один WOW-эффект затянется — сдвинется весь план

**Метрики контроля:**
- Все метрики из Сценария A +
- Landing engagement time: +15-20%
- Sample kit conversion: текущий → 25%
- Bounce rate landing: -10%

**Ожидаемый impact на конверсию:** +10-20%

### Сценарий C: Aggressive (полный effect rollout)

**Scope (недели 1-12):**

| Неделя | Задача |
|---|---|
| 1-2 | Lifecycle unification (ускоренный) |
| 3-4 | Compare + Infinite Cards + Vanish Input |
| 5-6 | Multi-Step Loader + Floating Dock + Sparkles |
| 7-8 | BG Beams + Dotted Glow + Text Generate |
| 9-10 | Tooltip Card + Animated Tooltip + Cover |
| 11 | Magnetic buttons + Stagger animations |
| 12 | Performance panic fix (всё тормозит 🔥) |

**Плюсы:**
- Максимальный визуальный эффект
- "Wow, этот сайт крутой!" первое впечатление
- Все эффекты из Effects.MD реализованы

**Риски:**
- 🔴 ВЫСОКИЙ риск: performance деградация
- 🔴 Shared hosting не выдержит тяжёлые JS
- 🔴 Regression bugs: новые эффекты ломают старые
- 🔴 CLS (Cumulative Layout Shift) penalty от Google
- 🔴 Mobile experience: тормоза на mid-range устройствах
- 🔴 QA coverage невозможна в 1 неделю для 12+ эффектов

**Метрики контроля:**
- Lighthouse Performance score: цель >70, реалистично 50-60
- Time to Interactive: цель <3s, реалистично 4-5s
- JS bundle size: цель <100KB, реалистично 150-200KB
- Mobile usability: 🔴 проблемы

**Ожидаемый impact на конверсию:** +5-10% или **-15%** если performance деградация. **Чистый risk/reward = отрицательный.**

## 7. Answers to Specific Questions

### 1. Compare autoplay: Включать по умолчанию или по hover?

**Ответ: Autoplay по умолчанию, но с нюансами.**

**Обоснование:**
- 60% аудитории — новички. Они не знают что слайдер интерактивный. Без autoplay они увидят статичную картинку и пролистают мимо.
- Autoplay привлекает внимание и демонстрирует ценность продукта без user action.

**Реализация:**
- Autoplay включается при входе в viewport (IntersectionObserver)
- Oscillation: 20% → 80% → 20% с easing, пауза 1.5 сек на краях
- При ЛЮБОМ user interaction (hover/touch/keyboard) — autoplay останавливается навсегда для этого экземпляра
- На mobile: autoplay по умолчанию (hover невозможен)
- `prefers-reduced-motion: reduce` → autoplay OFF, статичное 50% положение
- В "Pro mode" → autoplay OFF

### 2. Infinite Cards speed: какой оптимален для trust?

**Ответ: `normal` (30 секунд полный цикл) с адаптацией.**

**Обоснование:**
- **Slow (60 сек):** Пользователь может подумать что ничего не двигается. Но создаёт ощущение спокойствия и надёжности.
- **Normal (30 сек):** Заметное движение, успевает прочитать 1-2 карточки в viewport. Оптимальный баланс.
- **Fast (15 сек):** Нервное ощущение, не успевает прочитать. Ассоциация с "спешкой" подрывает trust.

**Рекомендация:**
- Desktop: `normal` (30 сек) — viewport шире, больше карточек видно одновременно
- Mobile: `slow` (45 сек, компромисс) — viewport уже, нужно больше времени на чтение одной карточки
- Hover pause: обязательно
- `data-speed` атрибут для A/B тестирования

### 3. Preflight WARN: Как уменьшить тревожность без скрытия проблем?

**Ответ: трёхуровневая система копирайта + визуальная демонстрация.**

**Фреймворк:**

```
Уровень 1: Заголовок (что) — НЕЙТРАЛЬНЫЙ
"Разрешение ниже рекомендуемого"

Уровень 2: Пояснение (почему это WARN, а не FAIL) — REASSURING
"Ваш файл 150 DPI. Мы рекомендуем 300 DPI для идеального результата,
но 150 DPI допустимо для принтов размером до 10x10 см."

Уровень 3: Действие (что делать) — EMPOWERING
"✅ Продолжить с текущим файлом"
"📐 Уменьшить размер принта для лучшего качества"
"📤 Загрузить файл с более высоким разрешением"
```

**Визуальная демонстрация:**
- При DPI WARN: показать side-by-side preview: "Как будет выглядеть ваш принт" (blurred) vs "Рекомендуемое качество" (sharp)
- Это Compare slider, встроенный в preflight результат!

**Дополнительно:**
- Иконки: ✓ (зелёная), ⚠ (жёлтая), ✗ (красная) — интуитивно понятны
- Не использовать слово "ошибка" для WARN → "рекомендация"
- Кнопка "Продолжить" должна быть ЗАМЕТНОЙ (primary style) — показать что WARN не блокирует

### 4. FAB vs Dock: Оставить одно или унифицировать?

**Ответ: Унифицировать в один Floating Dock, убрать FAB и отдельную Mobile Dock.**

**Обоснование:**
- 3 навигационных элемента в одном месте = UX хаос
- FAB — один action. Dock — множественная навигация. Dock включает в себя функционал FAB.
- Mobile Dock отдельным компонентом — дублирование кода и стилей.

**Реализация:**
```
Desktop:
  Floating Dock (bottom-center) с magnification
  Содержит: Home, Catalog, Prices, [ORDER] (primary CTA), Cabinet
  FAB → удалён, его action = primary CTA в Dock

Mobile:
  Fixed bottom bar (тот же компонент, другой layout)
  Без magnification (нет hover на touch)
  Тот же набор items
  Mobile Dock → удалена как отдельный компонент

z-index: --z-dock: 1000 (единственный floating nav)
```

### 5. 3D effects: Стоят ли performance trade-off?

**Ответ: НЕТ для текущего состояния проекта.**

**Обоснование:**
- 3D (CSS `perspective`, `transform3d`) на лёгком уровне — допустимо (Compare с subtle tilt)
- 3D (Three.js, WebGL) — нет. Bundle size, GPU load, battery drain, shared hosting
- Целевая аудитория: 60% новичков, которые хотят "понять и заказать", а не "wow 3D"
- Performance budget: каждый 3D эффект = -10-20 Lighthouse points

**Исключение:**
- CSS-only `perspective` + `rotateY` для карточек в каталоге — это дёшево и эффектно
- Пример: card hover → subtle 2-3° tilt. Чистый CSS, никакого JS.

### 6. Haptics/Sound: Для DTF B2B/B2C уместно?

**Ответ: Haptics — осторожно ДА. Sound — категорически НЕТ.**

**Haptics (Vibration API):**
- Поддержка: Android Chrome ~80%, iOS Safari = НЕТ
- Где уместно: успешная отправка заказа (короткая вибрация 50ms)
- Где НЕ уместно: навигация, hover, scroll, ошибки
- Реализация: `if ('vibrate' in navigator) navigator.vibrate(50);` — 1 строка
- Вывод: можно добавить как micro-delight, но не критично

**Sound:**
- Авто-звук = #1 раздражитель в web
- B2B клиенты открывают в офисе → звук = немедленное закрытие
- Даже opt-in звук: никто не включит
- Вывод: НЕТ. Никогда. Ни для какого сценария.

### 7. Multi-Step Loader: Fake честный или ждать infra fix?

**Ответ: Sync preflight СЕЙЧАС. Infra fix — отдельный проект.**

**Обоснование:**
- Celery/Redis на shared hosting — нерешаемо без миграции на VPS
- Fake multi-step loader — обман пользователя. "Проверяю DPI..." (setTimeout 260ms) → пользователь видит что это мгновенно. Trust подрывается.
- Sync preflight для файлов до 50MB — ≤2-3 секунды на backend. Вполне терпимо.

**Реализация:**
```
1. Пользователь загружает файл
2. JS: показывает spinner + "Анализируем ваш файл..."
3. POST /api/preflight/check/ (FormData)
4. Django view: синхронно запускает engine.py → JSON response
5. JS: получает результат → рендерит inline feedback
6. Если >5 сек — показать "Файл большой, это может занять ещё немного"
```

**Когда переходить на async:**
- Если средний файл >20MB
- Если нужен batch upload (10+ файлов)
- Если переедете на VPS → тогда Celery + WebSocket/SSE для real-time progress

### 8. Dark mode: Приоритет для DTF аудитории?

**Ответ: Низкий приоритет. Не делать до завершения Сценария B.**

**Обоснование:**
- Print-сервис: пользователь оценивает как будет выглядеть принт на ТКАНИ (обычно белой/чёрной). Preview на белом фоне — must have.
- Dark mode удваивает CSS работу: каждый цвет, каждый shadow, каждый border → нужен тёмный вариант.
- 60% новичков не используют dark mode в браузере (системная настройка, не web-preference).
- B2B клиенты: корпоративные устройства, часто нет dark mode.

**Если всё же делать (фаза 3+):**
- UI chrome (nav, footer, sidebar) → dark
- Preview/upload зона → ВСЕГДА белый фон
- Каталог карточки → белый фон для принтов
- Использовать CSS custom properties для токенов цвета

### 9. Motion limits: Сколько анимаций на страницу максимум?

**Ответ: Tiered system.**

| Tier | Тип | Лимит | Примеры |
|---|---|---|---|
| CSS-only | `transform`, `opacity`, `clip-path` transitions | Без лимита | Hover effects, Dock magnification, fade-in |
| JS-light | IntersectionObserver → add class | Max 8 на страницу | Encrypted Text, Stateful Button, fade-on-scroll |
| JS-medium | requestAnimationFrame loop (в viewport) | Max 3 одновременно | Compare autoplay, Infinite Cards |
| JS-heavy | Canvas, WebGL, complex DOM manipulation | Max 1 на страницу | (пока нет таких, и не нужно) |

**Enforcement:**
```javascript
DTF.animationBudget = { light: 0, medium: 0, heavy: 0 };
DTF.registerEffect = function(name, selector, init, tier = 'light') {
  const limits = { light: 8, medium: 3, heavy: 1 };
  if (DTF.animationBudget[tier] >= limits[tier]) {
    console.warn(`[DTF] Budget exceeded for ${tier}: ${name} not initialized`);
    return;
  }
  DTF.animationBudget[tier]++;
  // ... init
};
```

### 10. Mobile-first: Какие effects обязательны на mobile?

**Ответ:**

| Эффект | Mobile обязателен? | Обоснование |
|---|---|---|
| Compare Slider | ✅ ДА (tap-to-toggle mode) | Главный trust element. На mobile: tap переключает до/после |

| Floating Dock (bottom bar) | ✅ ДА | Основная навигация |

| Sparkles | ❌ НЕТ | Battery drain, отвлекает |
| Text Generate | ⚠️ Упрощённый | Без побуквенного typing — просто fade-in слов |
