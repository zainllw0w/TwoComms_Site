# TwoComms — глубокий разбор AEO/GEO/SEO и ревизия аудита Chyzh Agency

**Дата:** 2026-06-08
**Объект:** `https://twocomms.shop` (без поддоменов: dtf./management./fin./storage. исключены)
**Источник для ревизии:** PDF-аудит Chyzh Agency «AEO/GEO Аудит», 7 червня 2026, GEO Score 75/100
**Метод:** проверка каждого вывода аудита против живого сайта (curl), кодовой базы Django и боевой БД (SSH). Всё ниже — либо подтверждено фактически, либо помечено как гипотеза.

---

## 0. TL;DR (executive summary)

1. **Аудит Chyzh — шаблонный и в значительной части неверен/устарел.** Минимум 6 из его «находок» уже закрыты на сайте (OG-картинка, canonical, FAQ/Article/Product/AggregateRating schema, доступ AI-ботов). Главный «критический» тезис аудита — «низький бал ChatGPT через блокування GPTBot» — **фактически ложный**: `robots.txt` явно разрешает GPTBot и ещё 10 AI-ботов. Аудит даже сам себе противоречит (в его же таблице «GPTBot — Доступ є»).
2. **Реальный технический уровень сайта высокий.** Богатый schema-граф со стабильными `@id`, reciprocal hreflang, llms.txt + llms-full.txt, Speakable, HowTo, sitemap-индекс из 8 секций, полностью заполненные SEO-поля и alt-тексты по всем 65 товарам, **завершённый перевод RU/EN** (65/65 по title и full_description).
3. **Реальные слабые места — контентные и точечные, а не технические.** Главные дыры: пустой блог (1 пост), тонкая главная (~426 слов в `<main>`), висячая schema-ссылка `founder`, грамматически неверные автозаголовки, отсутствие entity-сигналов (Wikidata/GBP).
4. **Куда вкладывать усилия для AEO/GEO:** наполнение блога нишевым контентом, founder/Person entity, TL;DR-факт-блоки above-the-fold, расширение support/FAQ, консолидация Organization-графа, видимый `dateModified`. Это даст рост цитируемости в ChatGPT/Perplexity/AI Overviews сильнее, чем любые правки из PDF.

## 2. Реальные находки (верифицированы), которых нет в аудите или они искажены

### 2.1. ❗P0 — Висячая schema-ссылка `founder` (E-E-A-T / entity)

В `storefront/seo_utils.py` (`generate_organization_schema`, ~стр. 1495):

```python
"founder": {"@id": f"{base_url}#founder"},
```

**Нигде не эмитится Person-нода с `@id` = `https://twocomms.shop/#founder`.** Ссылка не резолвится → Google Knowledge Graph и AI-движки не могут связать бренд с основателем. При этом в `llms-full.txt` founder уже описан текстом (Артем Синіло, ветеран, Харків) и есть авторитетный внешний источник (пресрелиз mva.gov.ua), т.е. данные для entity ЕСТЬ, но не оформлены в schema.

**Что сделать:** эмитить `Person`-ноду (глобально или на `/pro-brand/`):
```json
{
  "@type": "Person",
  "@id": "https://twocomms.shop/#founder",
  "name": "Артем Синіло",
  "jobTitle": "Засновник TwoComms",
  "worksFor": {"@id": "https://twocomms.shop/#organization"},
  "nationality": "UA",
  "sameAs": ["<mva.gov.ua пресреліз>", "<1+1/Громадське сюжети>", "<особистий профіль, якщо є>"]
}
```
Это закрывает «named expert» из аудита и даёт самый сильный E-E-A-T-якорь для ниши «ветеранський бренд».

### 2.2. ❗P0 — Грамматически неверные автозаголовки (бьёт по AEO-цитируемости)

На живой PDP (`/product/classic-tshirt/`) H2 рендерятся так:
- `Доставка Футболка класична по Україні`
- `Догляд за Футболка класична`

Название товара вставляется в **именительном падеже** без морфологии. Корректно: «Догляд за футболкою «Класична»», «Доставка по Україні: футболка «Класична»». AI-движки цитируют именно H2/первые предложения секций — неграмотный украинский снижает вероятность цитирования и доверие, плюс это видят живые покупатели.

**Что сделать:** в генераторе `product_seo_block` (и шаблоне PDP) перестроить шаблоны заголовков так, чтобы название было обособлено (кавычки/двоеточие), а не склонялось внутри предлога. Не пытаться склонять программно (хрупко) — менять конструкцию фразы.

### 2.3. ⚠️P1 — Три параллельных `OnlineStore`-узла с рассинхроном

В графе сосуществуют:
- `#organization` — тип `["Organization","OnlineStore"]`, `availableLanguage: ["uk","ru","en"]`;
- `#storefront` (homepage) — `OnlineStore` + `AggregateOffer`;
- `#online-store` (тег `online_store_schema`, legacy) — `OnlineStore`, `availableLanguage: ["uk","ru"]`, **хардкод** `base_url = "https://twocomms.shop"` без учёта `SITE_BASE_URL`.

Три узла типа OnlineStore с разными `@id` и расходящимися полями (язык поддержки, contactPoint) размывают entity-картину для краулеров. `online_store_schema` дублирует то, что уже несёт `#organization`.

**Что сделать:** оставить `#organization` (брендовый entity) + `#storefront` (коммерческий узел главной с AggregateOffer). Узел `#online-store` вывести из использования или слить в `#storefront`. Синхронизировать `availableLanguage` и убрать хардкод URL.

### 2.4. ⚠️P1 — Sitemap товаров: 195 = 65×3 локали

`sitemap-products.xml` отдаёт 195 `<loc>` (по проверке) — это 65 товаров × 3 локали (i18n alternates восстановлены в Phase 2). **Это не баг** (переводы реально заполнены, см. §3.4), но:
- комментарии в коде (`sitemaps.py`, `views/static_pages.py`) всё ещё говорят про «65» — устарели, вводят в заблуждение;
- 178 URL в `sitemap-product-variants.xml` (self-canonical вариации цвет/крой) — следить, чтобы variant-страницы не были thin-дубликатами (отличие только цветом). Если уникального контента на варианте нет — Google может пометить near-duplicate.

**Что сделать:** обновить комментарии; для variant-URL обеспечить хотя бы 1–2 уникальных предложения (цвет/сценарий), либо вернуть часть на canonical→base, если контент не дифференцируется.

### 2.5. ❗P0/P1 — Тонкая главная, нет AI-TL;DR

Контент внутри `<main>` главной ≈ **426 слов** (совпадает с оценкой спеки US-7 «391 слово»). Нет факт-блока (TL;DR), который AI-движки берут «как есть». Первый абзац после H1 — то, что чаще всего цитируется.

**Что сделать (US-7):** 600–800 слов уникального текста + TL;DR из 4–5 само-достаточных фактов-предложений (бренд, Харків, ветеранське коріння, асортимент, ціновий діапазон, доставка), без маркетингових кліше.

### 2.6. ❗P1 — Тонкие support-страницы

`/doglyad-za-odyagom/` ≈ 405 слов в `<main>` (цель спеки US-4 — 800–1200). HowTo-schema там уже есть (8 вхождений) — это сильно, но текстовое тело короткое. Аналогично для /delivery/, /faq/, /wholesale/, /cooperation/.

**Что сделать:** довести до целевого объёма по US-4; FAQ — до 25–30 Q&A под реальные нишевые вопросы.

### 2.7. ⚠️P2 — Локализованные og:image могут быть копиями uk

В `i18n_links.py` честно описано: `social-preview-ru.jpg` / `-en.jpg` могут быть байт-идентичны uk, пока не выполнен рендер-пайплайн (нужен libcairo на сервере). Деградация безопасная, но US-17 недореализован — RU/EN соцпревью без локализованной типографики.

## 3. Точки роста для AEO / GEO (сверх аудита) — что реально усилит цитируемость

### 3.1. Entity-сигналы (самый высокий ROI для GEO)
- **Wikidata**: создать сущность TwoComms (UA + EN labels), указать `instance of: brand/clothing company`, `country: Ukraine`, `inception`, `founder`, `official website`. Затем добавить Wikidata URI в `Organization.sameAs`. Это резко повышает шанс попасть в Knowledge Panel и в ответы Gemini/Perplexity.
- **Google Business Profile** (даже без витрины — как онлайн-бренд/виробник у Харкові) → согласованный NAP с `Organization.address` (UA-63, Харків) и `contactPoint` (+380966543212).
- **Расширить `sameAs`**: сейчас минимальный набор (Instagram + Telegram). Добавить подтверждённые: пресреліз mva.gov.ua, сюжеты 1+1/Громадське, YouTube, TikTok, маркетплейс-витрины (Rozetka/Kasta — фиды уже есть в urls). Каждый authoritative `sameAs` — это GEO-якорь.

### 3.2. Блог как editorial hub — критичная контентная дыра
БД показывает **1 опубликованный пост** при том, что `llms.txt` и `llms-full.txt` позиционируют `/blog/` как «canonical editorial hub for news, product reviews and useful apparel guides». AI-движки переходят по этой ссылке и не находят контента → нет topical authority, нет свежих цитируемых пассажей.

**Что сделать:** 10–15 статей под информационные/нишевые запросы (DTF-друк vs шовкографія; як обрати розмір худі; догляд за принтом; streetwear коди; ветеранський бренд — історія; розмірні гайди). Каждая — с Article+Speakable, датою, автором (Person `#founder` або редакція), FAQ-хвостом. Это главный драйвер AEO на горизонте квартала.

### 3.3. Above-the-fold факт-блоки (TL;DR) на всех ключевых типах страниц
PDP, категории, support — первый блок после H1 должен содержать 3–5 атомарных фактов в формате «утверждение = самодостаточное предложение». AI-экстракторы предпочитают пассажи 200–400 слов с чёткими H2/H3 (это и аудит отметил как «RAG chunk readiness»). Сейчас PDP-секции есть (US-3 Phase A), но их надо довести по объёму/уникальности и грамматике (§2.2).

### 3.4. Использовать уже завершённый перевод
Переводы RU/EN заполнены на 100% (65/65 по title и full_description) — это сильный, недоиспользованный актив. Убедиться, что:
- внутренние ссылки на `/ru/` ведут на `/ru/` (US-6 CP-6.2, no cross-locale leak);
- schema `inLanguage` корректно меняется (уже реализовано через `_resolve_inlanguage_code`).

### 3.5. Freshness-сигналы
В Product schema уже есть `releaseDate`/`dateModified`. Добавить **видимый** маркер «Оновлено: {дата}» в текст PDP/статей (US-3 CP-3.6) — Perplexity и AI Overviews учитывают свежесть, а видимая дата усиливает машинный сигнал.

### 3.6. ImageObject с лицензией/caption
Для image-citation (Google Images, AI с картинками) — у `Organization.logo` уже есть `ImageObject` с caption. Распространить подход на ключевые product-изображения: `creditText`/`license`/`acquireLicensePage` усиливают шанс показа в визуальных ответах.

### 3.7. Дёшево «для галочки» (P1/P2, низкий риск)
- `/.well-known/ai.txt` (Allow для основных AI-ботов) — закрывает пункт аудита, минута работы.
- `/ai/summary.json`, `/ai/faq.json`, `/ai/service.json` — структурный JSON-дайджест бренда (можно генерить из тех же данных, что и llms-full.txt).
- RSS-фид для блога + `<link rel="alternate" type="application/rss+xml">` (после наполнения блога) — помогает AI-агрегаторам отслеживать свежесть.

## 4. Что проверить отдельно (открытые гипотезы, требуют полного crawl)

1. **Битые ссылки в HTML-контенте** (описания товаров, тело статей, CTA блога `cta_url`). Точечная проверка nav/footer чистая, но аудит говорит про 3 шт. — нужен Screaming Frog / собственный crawler по `<a href>` внутри `content_html` и `full_description`.
2. **Variant-страницы (178 URL)** — выборочно убедиться, что контент дифференцируется, иначе риск near-duplicate.
3. **Реальная keyword density** по топ-товарам — измерить, прежде чем что-либо «снижать» (не ломать осмысленный текст ради догмы 1.5%).
4. **rel=prev/next** на пагинации каталога — есть `rel="next"`, проверить `rel="prev"` на страницах >1 (некритично, Google их игнорирует, но консистентность не повредит).
5. **CWV на проде** (LCP/INP/CLS mobile) — техническая база сильная (отложенная аналитика, preload шрифтов, inline critical CSS), но стоит снять Lighthouse на топ-15 страницах и зафиксировать baseline.

## 6. Чего НЕ делать (чтобы не навредить)
- **Не «разблокировать GPTBot»** — он уже разрешён; не трогать рабочий `robots.txt` по совету аудита.
- **Не плодить `ai.txt` как замену `llms.txt`** — llms.txt уже полноценный; ai.txt максимум дополнение.
- **Не удалять `display:none`/`visually-hidden` из публичных шаблонов** — это функциональные UI-элементы и accessibility (`sr-only`), а не клоакинг.
- **Не «снижать плотность ключей» вслепую** — сначала измерить; ломать осмысленный текст ради 1.5% вредно.
- **Не схлопывать RU/EN** — они полностью переведены и приносят отдельный трафик; hreflang кластеризует корректно.

## 7. Итоговая оценка
Технически TwoComms — **сильный сайт уровня выше среднего по украинскому e-commerce** (богатый валидный schema-граф, полный i18n, llms.txt/full, Speakable, HowTo, заполненные SEO-поля и переводы). Аудит Chyzh это недооценивает и фокусируется на ложных/закрытых пунктах. Реальный потолок роста для AEO/GEO лежит **в контенте и entity-сигналах**, а не в технике: наполнить блог, оформить founder/Wikidata entity, добавить TL;DR-факт-блоки, расширить support/FAQ, починить два точечных бага (founder-нода, грамматика заголовков). Эти шаги дадут рост цитируемости в AI-поиске и позиций по нишевым кластерам, тогда как механическое следование PDF-аудиту — нет.

## 2.1. 🔴 P0 — «Недопустимый тип объекта в поле `parent_node`» (КРИТ, обнаружено 07.06.2026)

**Затронуто:** `/product/twocomms-reality-bends-dark-neon-edition/`, `/product/twocomms-reality-bends-future-2026/`.

**Корневая причина (подтверждена на живом JSON-LD):** на канонической PDP в одном `@graph` эмитятся **две ноды с одинаковым `@id`**:

1. standalone полный `Product`:
   `@id = https://twocomms.shop/product/.../#product`
2. минимальная Product-нода внутри `ProductGroup.hasVariant[0]` — **тот же** `@id = .../#product`.

```jsonc
// ProductGroup.hasVariant[0] — КОЛЛИЗИЯ:
{ "@type":"Product", "@id":".../twocomms-reality-bends-dark-neon-edition/#product", ... }
// и параллельно standalone Product с тем же @id в начале @graph
```

Google при сборке дерева вариантов видит, что «родительский узел» (`parent_node` = ProductGroup → его вариант) ссылается на `#product`, который одновременно является самостоятельным `Product`. Тип родителя/потомка в графе становится неоднозначным → **invalid object type for `parent_node`**.

**Важно:** это **регрессия**, внесённая фиксом от 2026-06-04 (`generate_product_group_schema`), который добавил inline-вариант-ноды, переиспользовав `#product`. Тот фикс закрыл ошибку D (offers), но породил A.

**Фикс (в `storefront/seo_utils.py`, `generate_product_group_schema`):**
- НЕ переиспользовать `#product` для базовой ноды внутри `hasVariant`. Варианты должны иметь собственные `@id` (для базовой — например `#variant-base` или просто не включать базовую ноду, а перечислять только реальные цвет/крой-варианты).
- На каждой вариант-ноде добавить обратную связь `"inProductGroupWithID": product.slug` (или `"isVariantOf": {"@id": group_id}`), как в документированном Google-паттерне вложенного ProductGroup.
- Убедиться, что `ProductGroup.@id` (`#product-group`) и `Product.@id` (`#product`) — разные (это уже так), и что ни одна вариант-нода не дублирует `#product`.

**Проверка после фикса:** Rich Results Test на обоих URL → 0 ошибок «parent_node»; затем «Проверить исправление» в GSC.

## 2.2. 🔴 P0 — «Отсутствует поле `size`» (КРИТ, обнаружено 07.06.2026)

**Затронуто:** те же 2 товара reality-bends.

**Причина:** standalone `Product` имеет top-level `size = "S, M, L, XL, XXL"` (подтверждено), **но** inline Product-ноды внутри `ProductGroup.hasVariant` содержат только `@id / name / url / image / sku / offers / color` — **без `size`**. Google Merchant требует `size` на каждом варианте apparel.

**Фикс:** в `_variant_offer`/построении вариант-ноды (`generate_product_group_schema`) добавить в каждую вариант-Product-ноду:
```python
node["size"] = ", ".join(resolve_product_sizes(product))   # или per-variant размеры
```
Опционально продублировать `material`, `gtin` (когда появится) для полноты Merchant-вариантов.

## 2.4. 🟡 P2 — «Отсутствует `aggregateRating` / `review`» (7 + 7, WARNING)

Это **предупреждения**, не ошибки: товары без одобренных отзывов не получают `aggregateRating`/`review` (порог — ≥1 одобренный отзыв, и это правильно — фейковить рейтинг нельзя). Google всегда подсвечивает отсутствие рейтинга у товаров.

**Что делать:** не «фиксить» в коде, а **набирать реальные отзывы** (механика отзывов уже есть: `reviews` app, модерация, `aggregateRating` появляется автоматически). Тактика: e-mail/Telegram-запрос отзыва после доставки, импорт отзывов с маркетплейсов (если есть). Это закроет 7 товаров и заодно даст звёзды в SERP.

## 2.5. 🔴 P0 (быстро) — `/push/unsubscribe/` отдаёт 405 → «Страница заблокирована из-за ошибки 4xx»

**Факт:** `GET /push/unsubscribe/` → **405 Method Not Allowed** (эндпоинт POST-only для web-push отписки). Google трактует 4xx как блокировку; страница попала в отчёт (1 URL).

**Причина:** маршрут не закрыт в `robots.txt` (там нет `/push/`), и Googlebot ходит на него GET-ом.

**Фикс (любой из):**
1. Добавить в `robots.txt` `Disallow: /push/` (быстрее всего; страница не имеет SEO-ценности). В коде `robots_txt` / `ROBOTS_INTERNAL_DISALLOW_PATHS` добавить `"/push/"`.
2. Либо отдавать на `GET` 200 с минимальной информационной страницей отписки (если ссылка шлётся пользователям в push) или 302 на главную.

Рекомендация: **вариант 1** (robots Disallow `/push/`) + опционально `noindex` на странице. Это убирает 4xx из индекс-отчёта без риска.

## 2.6. 🟠 P1 — Старые URL отдают 404 вместо 301 (потеря ссылочного веса)

**Факт из БД:** слагов с ведущим дефисом / в верхнем регистре / числовых в текущей БД **НЕТ** (проверено: `SLUGS_LEADING_HYPHEN=[]`, `UPPERCASE=[]`, `NUMERIC=[]`). Значит URL из примеров GSC (`/product/-death-gbs-ass-ts/`, `/product/-20-twocomms-Legend/s/`, `/product/105/`) — **исторические**: товары были переименованы, а Google добивает старые адреса по кэшу.

**Проблема:** `/product/105/` → **404** (а не 301 на актуальный слаг). Каждый такой 404 теряет накопленный ссылочный вес и историю кликов (отмечено и в `VILNI_AUDIT_DEEP_REVIEW`: «втрата історичних кліків через 301-ланцюг»).

**Фикс:**
- Завести карту соответствий «старый slug/id → новый slug» и отдавать **301** (через `django.contrib.redirects` — приложение уже подключено, есть `SubdomainRedirectFallbackMiddleware`).
- Для id-based URL (`/product/<id>/`) — добавить view-резолвер: если пришёл числовой сегмент и товар найден по id → 301 на канонический `/product/<slug>/`.
- Проверить полный список 404 в GSC (Отчёт «Не найдено (404)») и закрыть массово.

## 2.7. 🟠 P1 — Индексируются эфемерные ресайзы `/media/.../optimized/*_640w.avif`

**Факт:** URL `/media/products/extra/optimized/0BD45B49-..._640w.avif` попал в примеры GSC, а сейчас отдаёт **404**. Похоже, `ImageOptimizationMiddleware` генерирует ресайзы on-the-fly, их URL нестабильны → Google индексирует картинку, потом она исчезает (404 + crawl-шум + потеря image-SEO).

**Что проверить/сделать:**
- Убедиться, что в `sitemap-images.xml` отдаются **стабильные** image-URL (а не временные `_640w.avif`).
- Поставить `X-Robots-Tag: noindex` на ответы оптимизатора `/media/.../optimized/` (или отдавать стабильные кэшируемые файлы с длинным `Cache-Control`).
- Проверить, что ресайз-файлы реально персистятся, а не удаляются (иначе ломается и open-graph/image-индексация).

## 2.8. 🟠 P1 — Видео для Merchant / Rich Results (явный вопрос)

**Факт:** видео есть только у **1 товара** (id 104, reality-bends-dark-neon, `youtube.com/watch?v=nZRX318OvbI`). Сгенерированный `VideoObject`:
```json
{"@type":"VideoObject","name":"Відео огляд: ...","description":"...",
 "thumbnailUrl":"https://i.ytimg.com/vi/nZRX318OvbI/hqdefault.jpg",
 "contentUrl":"https://www.youtube.com/watch?v=nZRX318OvbI",
 "embedUrl":"https://www.youtube-nocookie.com/embed/nZRX318OvbI",
 "uploadDate":"2026-04-19"}
```

**Что сделано хорошо:** есть `name`, `description`, `thumbnailUrl`, `contentUrl`, `embedUrl`, `uploadDate` — базовый набор полей VideoObject присутствует, thumbnail валидный (ytimg), embed через nocookie.

**Проблемы:**
1. 🟠 **`video` — не валидное свойство `Product` в schema.org.** Сейчас VideoObject вложен как `Product.video`. У типа `Product` нет свойства `video` → Google его **игнорирует**: ни Video rich result, ни польза для Merchant. Корректно одно из:
   - вынести `VideoObject` **отдельной нодой** в `@graph` (eligible для Video rich results), или
   - привязать через `"subjectOf": {"@type":"VideoObject", ...}` на Product.
2. 🟠 **`uploadDate` фейковый** — берётся из `product.created_at` (2026-04-19), а не из реальной даты публикации видео на YouTube. Для VideoObject Google ожидает фактическую дату загрузки. Хранить отдельное поле/тянуть через YouTube oEmbed/Data API.
3. 🟡 `contentUrl` = watch-страница (Google предпочитает прямой media-файл; для YouTube надёжнее опираться на `embedUrl`, что уже есть — ок, но `contentUrl` на watch-URL формально слабее).

**Для Merchant-фида** (`google_merchant_feed.xml`): видео в товарном фиде Merchant не является сильным фактором; основная ценность видео — **Video rich result в Поиске**, который сейчас не работает из-за п.1. Поэтому приоритет — вынести VideoObject из `Product.video` в отдельную ноду/`subjectOf`.

**Фикс (`generate_product_schema`):** заменить
```python
schema["video"] = video_obj
```
на отдельную ноду графа (в `product_graph` добавить `video_obj` в `@graph`) или
```python
schema["subjectOf"] = video_obj
```
и подставить реальный `uploadDate`.

- GSC-предупреждения «Отсутствует aggregateRating / review» (по 7 в отчёте) фактически относятся ко **всем** товарам.
- Нет ни одной звезды в SERP → ниже CTR, чем у конкурентов с рейтингами.
- Механика отзывов (`reviews` app, модерация, авто-`aggregateRating` при ≥1 одобренном) **полностью готова, но не используется**.

**Что сделать (не фейк):** запустить сбор реальных отзывов — пост-доставочный запрос (e-mail/Telegram/в кабинете), импорт отзывов с маркетплейсов (Rozetka/Kasta, если есть), QR на упаковке. Даже 1 отзыв на товар включает звёзды и закрывает GSC-warning. Это самый высокий ROI из «контентных» правок: рейтинги + AEO-доверие.

## 3.5. 🟠 P1 — ProductGroup эмитится даже для одноцветных товаров

**Что сделать:** в `product_graph`/`generate_product_group_schema` эмитить ProductGroup **только когда есть реальная вариативность для индексации** (>1 цвет-варианта). Для одноцветных — оставить только полноценный `Product` (он и так rich-eligible). Это разом снижает площадь ошибок §2.1/§2.2 на ~58 товарах.

## 3.6. 🟡 P2 — Вечная «скидка» почти на всём ассортименте

**Риск:** перманентная скидка на ~97% ассортимента — это (а) сигнал «фейковой распродажи» для Google Merchant (могут урезать sale-аннотации/доверие), (б) обесценивание «скидки» как триггера, (в) потенциальные вопросы по закону о рекламе (зачёркнутая «старая» цена, которой не было). 

**Что сделать:** оставить скидки только там, где они реальны/срочны; для базовых товаров — обычная цена без `discount_percent`. Согласовать `priceValidUntil` с реальными промо-окнами, а не «+90 дней всегда».
